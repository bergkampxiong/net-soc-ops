# 部署与配置排查报告

本文档记录前端 HTTPS、前后端跨站/硬编码、数据库连接硬编码三项的排查结论，**在文档中给出实施修改方案**，并包含 **Docker 与数据库（PostgreSQL + Redis）安装方案及一键脚本**，便于按步骤落地。

---

## 一、问题与结论摘要

| 类别 | 是否硬编码/问题 | 主要位置 |
|------|----------------|----------|
| 前端 HTTPS | 未启用（开发常态） | 正式环境应在反向代理配置 |
| 前端请求后端 | 相对路径 `/api`，合理 | `request.ts` |
| 开发代理后端地址 | **硬编码 IP:端口** | `package.json`、`proxy-server.js` |
| 后端 CORS | 允许所有来源 `*`；同机同域部署时 CORS 不生效，**无需从环境变量读取**（见 2.2） | `main.py` |
| PostgreSQL | **硬编码** | `database/config.py` |
| Redis | **硬编码或默认硬编码** | `database/config.py`、`connection_pool_manager.py`、`redis_manager.py` |

---

## 二、实施修改方案（仅文档输出，按需执行）

以下为建议的修改步骤与示例，实际改代码时可按此在仓库中逐项实施。

### 2.1 前端：开发代理后端地址改为环境变量

**目标**：去掉 `172.18.40.99:8000` 硬编码，通过环境变量配置代理目标。

**步骤一：修改 `netops-frontend/proxy-server.js`**

- 在文件顶部增加从环境变量读取代理目标，例如：
  - 变量名：`REACT_APP_API_PROXY_TARGET`
  - 默认值：`http://127.0.0.1:8000`
- 将原先写死的 `target: 'http://172.18.40.99:8000'` 改为使用上述变量。
- 将日志或拼接中的 `targetUrl` 也改为基于该变量拼接，避免再次出现硬编码 IP。

**步骤二：新增前端环境示例文件**

- 在 `netops-frontend/` 下新增 `.env.development.example`（不提交敏感信息）：
  - 内容示例：`REACT_APP_API_PROXY_TARGET=http://127.0.0.1:8000`
  - 说明：使用 `npm run proxy` 或 `start-all` 时，代理目标由此变量决定；仅用 `npm run start` 时由 `package.json` 的 `proxy` 字段决定。

**步骤三：修改 `netops-frontend/package.json`**

- 将 `"proxy": "http://172.18.40.99:8000"` 改为本地默认，例如：`"proxy": "http://127.0.0.1:8000"`。
- 在项目 README 或本文档中说明：需要连其他后端时，使用 `proxy-server.js` 并配置 `REACT_APP_API_PROXY_TARGET`，或按需改 `proxy`（不推荐提交个人 IP）。

---

### 2.2 后端：CORS 与同机部署（**一般不必从环境变量读取**）

**结论**：前后端若在同一台设备上部署（常见做法），通常由同一反向代理对外提供同一域名（如 Nginx 静态资源 `/` + 接口 `/api` 转发到后端），浏览器请求为**同源**，不会触发 CORS 校验，因此**无需**为“正式环境限制 CORS 来源”而增加环境变量。

- **同机同域**：保持当前 `allow_origins=["*"]` 即可，无需修改。
- **仅当前后端分离到不同域名时**（例如前端 `https://app.example.com`、后端 `https://api.example.com`），才有必要限制来源；此时可再考虑从环境变量读取 `ALLOWED_ORIGINS`（逗号分隔域名），未设置时默认 `*`。

**建议**：不将“CORS 从环境变量读取”列为必做项；若未来确需前后端不同域部署，再按需在 `main.py` 中增加 `ALLOWED_ORIGINS` 的读取逻辑。

---

### 2.3 后端：数据库连接从环境变量读取

**目标**：去掉 `database/config.py` 中 PostgreSQL 的 IP/端口/用户/密码硬编码，优先使用环境变量。

**修改 `netops-backend/database/config.py`**

- **方案 A（推荐）**：优先使用完整连接串
  - 若已设置 `DATABASE_URL`，则直接使用，不再用硬编码覆盖。
  - 若未设置，再从 `DB_HOST`、`DB_PORT`、`DB_USER`、`DB_PASSWORD`、`DB_NAME` 等环境变量（或合理默认值，如 localhost）拼接 `DATABASE_URL`。
- **方案 B**：仅用分项环境变量
  - 删除硬编码的 `DATABASE_CONFIG` 字面量，改为从环境变量读取：如 `DB_HOST`（默认 `127.0.0.1`）、`DB_PORT`（默认 `5432`）、`DB_USER`、`DB_PASSWORD`、`DB_NAME`（默认 `netops`），再在 `get_database_url()` 中拼接。
- 无论哪种，末尾**仅在未设置时**写入 `os.environ["DATABASE_URL"]` 和 `os.environ["CMDB_DATABASE_URL"]`，避免覆盖已有 `.env` 配置。
- 默认值建议仅适用于本地开发（如 `127.0.0.1`、空密码等），不在仓库中写生产 IP 或真实密码。

---

### 2.4 后端：Redis 从环境变量读取

**目标**：统一 Redis 配置来源，去掉各处硬编码的 `172.18.40.80`。

**修改 1：`netops-backend/database/config.py`**

- `REDIS_CONFIG` 不再写死 host/port，改为从环境变量读取，例如：
  - `REDIS_HOST`（默认 `127.0.0.1`）
  - `REDIS_PORT`（默认 `6379`）
  - `REDIS_DB`（默认 `0`）
  - 若有 `REDIS_URL`，可直接用于 `get_redis_url()`，否则用上述分项拼接。
- 仅当未设置时再向 `os.environ["REDIS_URL"]` 写入，避免覆盖。

**修改 2：`netops-backend/utils/connection_pool_manager.py`**

- Redis 客户端构造时不再写死 `host='172.18.40.80'`。
- 改为从环境变量读取（如 `REDIS_HOST`、`REDIS_PORT`、`REDIS_DB`，或解析 `REDIS_URL`），默认值建议 `127.0.0.1`、`6379`、`0`。

**修改 3：`netops-backend/utils/redis_manager.py`**

- 当前已支持环境变量，仅将默认回退值由 `172.18.40.80` 改为 `127.0.0.1`，与上述一致。

---

### 2.5 前端 HTTPS（开发可选 + 生产反向代理）

**目标**：不在前端业务代码里写死证书或域名；证书与域名在反向代理层或开发环境可选开关中处理。

**已实现（增量，不修改现有功能）：**

- **开发环境可选 HTTPS**  
  - 通过环境变量 `REACT_APP_DEV_HTTPS=true` 启用（如 `REACT_APP_DEV_HTTPS=true npm run start`）。  
  - 若已生成长期有效自签名证书（执行 `bash scripts/gen-dev-https-cert.sh`，默认 10 年，输出到 `netops-frontend/.cert/`），devServer 会优先使用该证书；否则使用 webpack-dev-server 默认自签名。默认未设置时仍为 HTTP，行为与原来一致。  
  - **前端证书配置**：系统管理 → 证书配置（`/system` 页签）可配置「开发环境启用 HTTPS」与「自签名证书有效期（天）」并落库，与上述脚本及 devServer 行为对应。
- **生产环境**  
  - 使用 Nginx（或 Caddy / Traefik）在反向代理层配置 TLS。  
  - 示例配置已提供：**`scripts/nginx-https-example.conf`**。替换其中的 `server_name`、`ssl_certificate`/`ssl_certificate_key`、`root`（前端 build 目录）、`proxy_pass`（后端地址）即可。  
  - 同域：静态资源由 Nginx 提供，`/api` 转发到后端，前端继续使用相对路径 `baseURL: '/api'`，无需改请求逻辑。

**约定**：证书使用 Let’s Encrypt 或企业证书，在 Nginx 等代理配置中指定；不为前端构建或 devServer 配置证书文件路径。

## 三、实施顺序建议

1. **先做后端**：`database/config.py`、`connection_pool_manager.py`、`redis_manager.py`，避免继续依赖硬编码 IP/密码（CORS 同机同域时不必改）。
2. **再做前端**：`proxy-server.js`、`package.json`、`.env.development.example`，并更新 README 或本文档中的使用说明。
3. **最后**：在测试环境用环境变量验证数据库、Redis、代理目标，确认无误后再考虑生产变更。

## 四、环境变量汇总（便于部署与文档）

| 用途 | 变量名 | 示例/默认 | 说明 |
|------|--------|-----------|------|
| 前端开发 HTTPS（可选） | `REACT_APP_DEV_HTTPS` | `true` 启用 | 仅开发环境；启用后 devServer 使用自签名 HTTPS |
| 前端开发代理目标 | `REACT_APP_API_PROXY_TARGET` | `http://127.0.0.1:8000` | 仅 `proxy-server.js` 使用 |
| 后端 CORS 来源（可选） | `ALLOWED_ORIGINS` | `*` | 仅前后端**不同域**时需配置；同机同域部署无需 |
| 数据库连接串 | `DATABASE_URL` | — | 优先于分项 DB_* |
| 数据库分项 | `DB_HOST`, `DB_PORT`, `DB_USER`, `DB_PASSWORD`, `DB_NAME` | `127.0.0.1`, `5432`, … | 未设 DATABASE_URL 时使用 |
| Redis 连接串 | `REDIS_URL` | `redis://127.0.0.1:6379/0` | 可选 |
| Redis 分项 | `REDIS_HOST`, `REDIS_PORT`, `REDIS_DB` | `127.0.0.1`, `6379`, `0` | 未设 REDIS_URL 时使用 |

## 五、Docker 与数据库安装方案及脚本

适用于在本机或单机环境快速安装 Docker、PostgreSQL、Redis，并与项目配置（`database/config.py` / `.env`）一致，便于后端直接对接。

### 5.1 约定与前提

| 项目 | 说明 |
|------|------|
| 操作系统 | Linux（Debian/Ubuntu 或 CentOS/RHEL） |
| 磁盘 | PostgreSQL 约 **100G**、Redis 约 **50G**，**合计约 150G**；脚本会校验 **/app 所在分区**可用空间 |
| 持久化目录 | **/app**：PostgreSQL → `/app/netops-postgres-data`，Redis → `/app/netops-redis-data`（绑定挂载） |
| Docker 桥接 | 脚本会配置 **192.168.0.0/16**，默认桥 **192.168.0.1/25**，自定义网络 **/25** 子网（需 root 写 `/etc/docker/daemon.json`） |
| PostgreSQL | 用户 `amber`，密码 `amberman@2025!`，数据库 `netops`，端口 `5432` |
| Redis | 端口 `6379`，DB 0，当前无密码（可选后续加 requirepass） |

### 5.2 一键脚本

- **路径**：`scripts/setup-docker-databases.sh`
- **执行**：**建议 `sudo bash scripts/setup-docker-databases.sh`**（安装 Docker 与写 daemon.json 需 root）
- **前提**：`/app` 目录存在，且 `/app` 所在分区可用空间 ≥ 150 GiB（脚本会自动校验，不足则退出）；端口 5432、6379 未被占用。

脚本会依次：

1. **0** 检查 `/app` 存在并校验可用空间 ≥ 150 GiB，不足则退出；创建 `/app/netops-postgres-data`、`/app/netops-redis-data`。
2. **1** 若未检测到 Docker，按系统类型自动安装（Debian/Ubuntu 用 get.docker.com，CentOS/RHEL 用 yum）；写入 `/etc/docker/daemon.json`（桥接 192.168.0.0/16、/25），重启 Docker。
3. **2** 启动 PostgreSQL 容器 `netops-postgres`，数据挂载到 `/app/netops-postgres-data`。
4. **3** 启动 Redis 容器 `netops-redis`，数据挂载到 `/app/netops-redis-data`。
5. **4** 执行连接验证并输出示例 `DATABASE_URL`、`REDIS_URL`。

### 5.3 手动安装 Docker（脚本未安装时参考）

**Debian/Ubuntu：**

```bash
curl -fsSL https://get.docker.com -o get-docker.sh
sudo sh get-docker.sh
sudo usermod -aG docker "$USER"
```

**CentOS/RHEL/Fedora：**

```bash
sudo yum install -y yum-utils
sudo yum-config-manager --add-repo https://download.docker.com/linux/centos/docker-ce.repo
sudo yum install -y docker-ce docker-ce-cli containerd.io
sudo systemctl enable --now docker
sudo usermod -aG docker "$USER"
```

### 5.4 手动启动数据库（未使用脚本时）

PostgreSQL（数据目录 `/app/netops-postgres-data`）：

```bash
mkdir -p /app/netops-postgres-data
docker run -d --name netops-postgres --restart unless-stopped \
  -e POSTGRES_USER=amber -e POSTGRES_PASSWORD='amberman@2025!' -e POSTGRES_DB=netops \
  -p 5432:5432 -v /app/netops-postgres-data:/var/lib/postgresql/data postgres:16
```

Redis（数据目录 `/app/netops-redis-data`）：

```bash
mkdir -p /app/netops-redis-data
docker run -d --name netops-redis --restart unless-stopped \
  -p 6379:6379 -v /app/netops-redis-data:/data redis:7 redis-server --appendonly yes
```

### 5.5 与项目配置的对应关系

| 项目配置 | 本方案 |
|----------|--------|
| `database/config.py` 中 PostgreSQL | host 为本机 IP 或 `127.0.0.1`，port 5432，user=amber，password=amberman@2025!，database=netops |
| `database/config.py` 中 Redis | host 为本机 IP 或 `127.0.0.1`，port 6379，db 0 |
| `.env` 的 `DATABASE_URL` / `CMDB_DATABASE_URL` | `postgresql://amber:<URL编码密码>@<host>:5432/netops` |
| `.env` 的 `REDIS_URL` | `redis://<host>:6379/0` |

密码中的 `@` 在 URL 中应编码为 `%40`，例如：`postgresql://amber:amberman%402025!@127.0.0.1:5432/netops`。

## 六、尚未解决的问题

**说明**：前后端同机部署时，开发代理或请求地址写死为 `127.0.0.1:8000` **问题不大**，可接受；仅多环境或多人开发需连不同后端时再考虑改为环境变量（方案见 2.1）。

**数据库/Redis 连接**：已按 2.3、2.4 完成代码修改，`.env` 或环境变量中的 `DATABASE_URL`、`REDIS_URL`（或分项 `DB_*`、`REDIS_*`）会优先使用，不再被硬编码覆盖。可使用 **`scripts/install-netops.sh`** 交互输入数据库与 Redis 参数，由脚本写入 `netops-backend/.env` 并完成前后端依赖安装与库表初始化。
