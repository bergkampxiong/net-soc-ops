# 作业监控与报告 - 产品需求文档（PRD）

**文档版本**：1.0  
**关联文档**：`docs/作业执行控制_PRD.md`  
**菜单位置**：任务作业管理 → 作业监控与报告（`/rpa/task-job-management/job-monitoring`）

---

## 1. 背景与目标

### 1.1 背景

- **作业执行控制**（`/rpa/job-execution/jobs`）提供作业列表、执行/暂停/转定期等操作；点击作业名称进入**作业详情**，详情内包含「基本信息」与「**执行历史**」Tab，可查看该作业下的历次执行记录、日志、重试。
- **作业监控与报告** 当前为占位页（Empty），无实际功能；与「执行历史」相关的能力仅分布在作业详情内，缺少**跨作业、全局视角**的监控与汇总。
- 运维/管理员需要在一个页面内：查看**全部作业**的执行健康度、最近执行情况、快速定位失败执行并查看日志或跳转至对应作业，必要时做简单报告或导出。

### 1.2 产品目标

1. **全局执行监控**：在「作业监控与报告」页提供跨作业的执行概览（统计卡片 + 最近执行列表），不替代作业详情，而是与详情互补。
2. **统一执行列表**：支持按作业、状态、时间范围筛选的「最近执行」列表，支持查看日志、重试、跳转至作业详情。
3. **可选报告能力**：支持按周期汇总或导出执行记录（可作为一期或二期），便于审计与汇报。

### 1.3 与现有页面的边界

| 页面 | 职责 | 说明 |
|------|------|------|
| 作业执行控制（列表） | 作业的增删改查、执行/暂停/转定期等操作 | 不展示「单次执行」的全局列表；仅展示作业维度 |
| 作业详情（点击作业名称） | 单作业的基本信息、**该作业**的执行历史、单次执行的日志/重试 | 保留现有「执行历史」Tab；可增加「从监控页跳转」的入口 |
| **作业监控与报告** | **跨作业**的执行概览、最近执行列表、统计与报告 | 新建/完善本页；不替代作业详情，而是全局视角入口 |

### 1.4 用户角色

| 角色 | 说明 |
|------|------|
| 管理员/运维 | 查看执行概览与最近执行列表，定位失败执行、查看日志、重试或跳转至作业详情；使用报告/导出（若提供）。 |

## 2. 功能需求

### 2.1 监控概览（统计卡片）

| 需求 ID | 描述 | 验收标准 |
|---------|------|----------|
| M1.1 | 页面上方展示统计卡片，至少包含：今日执行次数、成功次数、失败次数、成功率（%） | 数据来源于 job_executions，按 start_time 所在日期聚合（今日 = 当前时区当日） |
| M1.2 | 可选：近 7 日 / 近 30 日执行趋势（柱状图或折线图） | 按日聚合执行次数（或成功/失败分层），图表可配置时间范围 |
| M1.3 | 统计支持按「作业」筛选（可选），未选时表示全部作业 | 前端筛选项与下方执行列表筛选联动（见 2.2） |

**涉及**：  
- 后端：提供 `GET /api/jobs/executions/stats` 或等价聚合接口（参数：date_from, date_to, job_id 可选）；或由前端多次调用现有接口在内存中聚合（若数据量小）。  
- 前端：`JobMonitoring.tsx` 顶部 Statistic 卡片 + 可选图表组件。

---

### 2.2 最近执行列表（跨作业）

| 需求 ID | 描述 | 验收标准 |
|---------|------|----------|
| M2.1 | 展示「最近执行」列表，列包含：执行 ID、关联作业名称、状态、开始时间、结束时间、执行结果摘要、错误信息（若有）、操作（查看日志、重试、跳转作业详情） | 数据来自 job_executions，需关联 jobs 表得到作业名称；按 start_time 倒序 |
| M2.2 | 支持分页（如每页 20 条），支持筛选：作业（下拉）、执行状态（成功/失败/执行中）、时间范围（开始时间） | `GET /api/job-executions` 或 `GET /api/jobs/executions` 支持跨 job 查询及上述筛选、分页 |
| M2.3 | 操作列：「查看日志」弹窗展示该次执行的 logs；「重试」调用现有 `POST /api/jobs/{job_id}/retry/{execution_id}`（若后端已实现）或 `POST /api/jobs/{job_id}/execute` 并提示；「跳转作业详情」进入 `/rpa/job-execution/jobs/{job_id}`，可选定位到执行历史 Tab | 与作业详情内「执行历史」的操作一致，避免重复实现逻辑 |

**涉及**：  
- 后端：新增 `GET /api/job-executions`（或扩展现有接口），支持 query：job_id, status, start_time_from, start_time_to, skip, limit；返回含 job_name 的执行列表。  
- 前端：Table + 筛选表单 + 查看日志 Modal + 重试/跳转逻辑。

---

### 2.3 与作业详情的联动

| 需求 ID | 描述 | 验收标准 |
|---------|------|----------|
| M3.1 | 从「作业监控与报告」的某条执行可跳转到对应作业详情页，并可选打开「执行历史」Tab | 跳转链接：`/rpa/job-execution/jobs/{job_id}?tab=executions`；JobDetail 支持通过 URL 参数 defaultActiveKey 打开执行历史 Tab |
| M3.2 | 作业详情页「执行历史」Tab 保持现有能力（表格、查看日志、重试），不删除、不弱化 | 仅增强入口（从监控页跳转）；作业详情仍为「单作业」视角的权威页面 |

---

### 2.4 报告与导出（可选 / 二期）

| 需求 ID | 描述 | 验收标准 |
|---------|------|----------|
| M4.1 | 支持「导出」当前筛选条件下的执行记录为 CSV 或 Excel | 后端提供 `GET /api/job-executions/export?format=csv&...` 或前端基于现有列表接口生成 CSV 下载 |
| M4.2 | 可选：按日/周/月汇总报告（成功数、失败数、成功率、失败原因分类） | 聚合接口 + 简单报告页或导出为 PDF/HTML；可列为二期 |

## 3. 数据与接口

### 3.1 现有数据

- **job_executions**：id, job_id, status, start_time, end_time, result, error_message, logs, created_at, updated_at。  
- **jobs**：id, name, process_definition_id, run_type, status, ...  
- 现有接口：`GET /api/jobs/{job_id}/executions` 仅支持按 job_id 查询单作业执行历史。

### 3.2 新增/扩展接口

**跨作业执行列表（必须）**

- **GET /api/job-executions**（建议路径，与 `/jobs/{id}/executions` 区分）  
  - Query：job_id（可选）, status（可选）, start_time_from, start_time_to（可选）, skip, limit  
  - 响应：`{ "total": number, "items": JobExecutionListItem[] }`  
  - `JobExecutionListItem`：在现有 JobExecution 字段基础上增加 `job_name`（或 job_id 由前端根据 job_id 再查名称，若后端不联表）。

**监控统计（必须）**

- **GET /api/job-executions/stats**  
  - Query：date_from, date_to（必填，日期），job_id（可选）  
  - 响应：`{ "total": number, "success": number, "failed": number, "running": number, "success_rate": number }`  
  - 或由前端用「执行列表」接口分页聚合（不推荐在数据量大时使用）。

**导出（可选）**

- **GET /api/job-executions/export?format=csv&...**  
  - Query 与列表接口一致，返回 CSV 文件流或 JSON 由前端转 CSV。

### 3.3 重试接口

- 若已有 `POST /api/jobs/{job_id}/retry/{execution_id}`，监控页「重试」直接调用。  
- 若未实现，可用「再次执行作业」：`POST /api/jobs/{job_id}/execute`，并在 UI 上标注为「再次执行」而非「重试该次」。

## 4. 前端规格

### 4.1 页面结构（作业监控与报告）

| 区域 | 内容 |
|------|------|
| 顶部筛选 | 时间范围（今日/近7日/近30日/自定义）、作业下拉（可选）、执行状态（可选）；与下方列表筛选联动 |
| 统计卡片 | 今日（或所选时间）执行总数、成功数、失败数、成功率；可选：近 7 日趋势图 |
| 最近执行列表 | 表格：执行 ID、作业名称、状态、开始时间、结束时间、结果摘要、错误信息、操作（查看日志、重试、跳转详情）；分页 |
| 底部可选 | 「导出」按钮（当前筛选结果导出为 CSV/Excel） |

### 4.2 路由与入口

- 路由：`/rpa/task-job-management/job-monitoring`（对应现有 `JobMonitoring` 组件）。  
- 菜单：任务作业管理 → 作业监控与报告（已有，当前为 Empty，替换为本 PRD 实现）。  
- 从作业详情「执行历史」可保留「返回监控」链接，跳回 `/rpa/task-job-management/job-monitoring`。

### 4.3 与作业详情的参数约定

- 跳转至作业详情并打开执行历史 Tab：`/rpa/job-execution/jobs/{job_id}?tab=executions`。  
- JobDetail 组件读取 `searchParams.get('tab')`，若为 `executions` 则 `defaultActiveKey="executions"`。

---

## 5. 非功能需求

- **性能**：执行列表默认分页（如 20 条/页）；统计与列表接口建议支持时间范围限制，避免全表扫描。  
- **权限**：与作业执行控制一致，仅管理员/运维可访问（具体依现有权限体系）。  
- **兼容**：不改变现有「作业执行控制」与「作业详情」的已有能力，仅新增/扩展接口与监控页实现。
