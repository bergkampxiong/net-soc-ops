# 作业执行控制 - 功能重构设计

## 一、现状与问题

### 1.1 当前实现概览

| 模块 | 路径/入口 | 功能 |
|------|-----------|------|
| 流程管理 | `/rpa/process-orchestration/process-management` | 流程列表、发布（草稿→已发布）、禁用、重新发布、编辑（仅草稿）、历史版本、删除 |
| 作业执行控制 | 菜单「任务作业管理 → 作业执行控制」→ `JobExecution` / `JobList` | 作业列表、**新建作业**、执行/暂停/恢复/终止、执行历史、编辑作业 |
| 作业后端 | `GET/POST/PUT/DELETE /api/jobs`, `POST /api/jobs/{id}/execute` 等 | Job 与 Process 无关联；`execute_job` 仅写执行记录，不跑流程代码 |

- **Job 模型**：`jobs` 表含 name、description、job_type（network_config/device_check/data_collection）、status、parameters、schedule_config 等，**无 process_definition_id**。
- **流程发布**：仅更新 `process_definitions.status = 'published'`，**不自动创建 Job**。
- **执行作业**：`JobService.execute_job` 只创建一条 `JobExecution`、更新 `last_run_at`，Celery 任务 `execute_job_task` 为 TODO，未真正执行流程或生成代码。

### 1.2 与配置管理模块的关系

- 流程设计器「配置备份」节点生成的代码中，会调用 **`POST /api/config-module/backups`** 写入备份（device_id、device_host、content、source=workflow、remark 等）。
- 配置管理模块的「配置管理」页（`/config-module/management`）通过 **`GET /api/config-module/backups`** 展示同一张表 `config_module_backups` 的数据。
- **结论**：只要运行生成的流程代码时能访问到同一后端（或配置正确的 `CONFIG_MODULE_API_URL`），备份数据会正常出现在「配置管理」中，**无需改配置管理功能**。需在部署与执行环境中保证 API 可达性及权限。

## 二、重构目标与原则

1. **作业仅来自发布**：作业执行控制中不再提供「新建作业」；作业仅由「流程发布」自动产生。
2. **发布即一次作业**：发布后的流程以「执行一次」的作业形态进入作业执行控制，需管理员手动触发执行。
3. **可转为定期作业**：管理员可将「一次作业」改为「定期作业」，并配置调度（如 cron/间隔）。
4. **明确区分一次 / 定期**：列表与操作上清晰区分「一次作业」与「定期作业」。
5. **执行即跑流程**：作业执行时真正运行该流程（生成代码执行或服务端解释执行），并将配置备份等结果写入配置管理模块（已满足）。

## 三、数据与接口设计

### 3.1 Job 模型扩展（建议）

在现有 `jobs` 表上增加或约定以下含义，必要时增加字段：

| 字段 | 类型 | 说明 |
|------|------|------|
| `process_definition_id` | VARCHAR(36), 可空 | 关联流程定义 ID；非空表示「由发布产生的流程作业」 |
| `run_type` | VARCHAR(20) | `once` = 一次作业，`scheduled` = 定期作业；默认 `once` |
| `schedule_config` | JSON | 仅当 run_type=scheduled 时有效；cron_expression、interval_seconds、timezone 等 |

约束与规则：

- 作业执行控制列表只展示 `process_definition_id IS NOT NULL` 的作业（即「由发布进来的流程」）。
- 新建作业接口（若保留给内部/管理员）仅允许在「发布流程」时由系统创建，不对外提供任意新建。

### 3.2 流程发布时创建作业

- **时机**：流程管理「发布」操作时（`POST /api/process-definitions/{id}/publish`）。
- **动作**：在 `process_definitions` 更新为 published 后，在 `jobs` 表中**新增一条作业**：
  - `process_definition_id = process_id`
  - `name` = 流程名称（或「流程名称 - 备份」等）
  - `job_type` = 如 `config_backup` 或沿用现有类型
  - `run_type` = `once`
  - `status` = `created`
  - `schedule_config` = null 或默认
- **幂等**：同一流程多次「发布」时，可约定仅创建一条作业（例如按 process_definition_id 唯一），或「重新发布」时更新已有作业的 name 等，避免重复多条。

### 3.3 作业执行控制接口行为

| 能力 | 说明 |
|------|------|
| 列表 | `GET /api/jobs` 增加筛选 `source=published` 或仅返回带 `process_definition_id` 的作业；支持按 `run_type` 筛选（一次 / 定期）。 |
| 禁止随意新建 | 前端移除「新建作业」入口；后端可保留 `POST /api/jobs` 但仅允许在发布流程时由内部调用，或仅接受带 `process_definition_id` 的创建。 |
| 执行 | `POST /api/jobs/{id}/execute`：根据 `job.process_definition_id` 取流程定义，生成代码并执行（或调用已有执行引擎），执行结果中配置备份会写入 `POST /api/config-module/backups`，自然落入配置管理。 |
| 转为定期 | 新增或复用 `PUT /api/jobs/{id}`：允许管理员将 `run_type` 从 `once` 改为 `scheduled`，并写入 `schedule_config`（cron 等）。 |
| 暂停/恢复/终止 | 仅对定期作业或运行中作业有意义，逻辑可保持现有设计。 |

## 四、前端交互设计

### 4.1 作业执行控制页（列表）

- **移除**：「新建作业」按钮及 `/rpa/job-execution/jobs/new` 路由（或仅对非「发布来源」隐藏）。
- **区分一次 / 定期**：
  - 方案 A：Tab 切换「一次作业」「定期作业」。
  - 方案 B：列表增加「运行类型」列，筛选器增加「运行类型：全部 / 一次 / 定期」。
- **列建议**：作业名称、关联流程（名称或 ID）、运行类型（一次 / 定期）、状态、上次执行时间、下次执行时间（定期）、操作。
- **操作**：
  - **一次作业**：执行、查看详情、**转为定期**（跳转或弹窗配置 schedule_config）。
  - **定期作业**：执行、暂停、恢复、终止、编辑（仅调度配置）、查看详情。

### 4.2 作业详情页

- 展示关联流程（名称 + 跳转流程设计器只读或管理页）。
- 一次作业：突出「执行」「转为定期」。
- 定期作业：展示调度配置、下次执行时间，并提供暂停/恢复/终止、编辑调度。

### 4.3 流程管理页

- 发布流程后，可提示「已发布，可在【作业执行控制】中执行或设为定期作业」。
- 可选：在流程列表增加「执行」快捷入口，跳转到对应作业详情或执行。

## 五、执行与配置管理闭环

### 5.1 执行链路

1. 管理员在作业执行控制点击「执行」→ `POST /api/jobs/{id}/execute`。
2. 后端根据 `job.process_definition_id` 获取流程定义（nodes/edges），调用现有 `CodeGenerator` 生成 Python 脚本。
3. 执行脚本：连接设备、执行配置备份/下发等；配置备份节点内调用 `POST /api/config-module/backups`。
4. 更新 `JobExecution` 状态与日志；作业 `last_run_at` 更新。

### 5.2 与配置管理（/config-module/management）的衔接

- 生成的流程代码已通过 **`POST /api/config-module/backups`** 写入 `config_module_backups` 表。
- 配置管理页通过 **`GET /api/config-module/backups`** 展示同一表，支持按设备、时间、关键词筛选及版本对比。
- **无需修改配置管理功能**；需保证：
  - 执行环境（运行生成代码的进程或执行引擎）能访问到与前端相同的 `CONFIG_MODULE_API_URL`（或同机默认 `http://127.0.0.1:8000/api`）。
  - 若跨机部署，需配置网络与鉴权（若 API 需要）。

### 5.3 检查清单（流程代码 ↔ 配置管理）

| 检查项 | 状态 |
|--------|------|
| 生成代码是否调用 POST /api/config-module/backups | 是（code_generator 中已实现） |
| 请求体是否包含 device_id, device_host, content, source=workflow, remark | 是 |
| 配置管理页是否从同一 backups 表读取 | 是（GET /config-module/backups） |
| 执行环境是否可访问配置模块 API | 需部署时配置 CONFIG_MODULE_API_URL |

## 六、实施要点小结

1. **作业执行控制不允许新建作业**：前端去掉「新建作业」，作业仅由「流程发布」创建。
2. **发布即生成一次作业**：发布时在 `jobs` 表插入一条 `run_type=once`、带 `process_definition_id` 的作业；管理员在作业执行控制中点击「执行」触发运行。
3. **管理员可改为定期作业**：提供「转为定期」操作，更新 `run_type=scheduled` 并写入 `schedule_config`；定期由调度器按配置触发执行（调度器实现可后续迭代）。
4. **区分一次与定期**：列表与筛选项明确「一次作业」「定期作业」，便于管理员识别与操作。
5. **配置管理**：当前生成的流程代码已能写入配置管理模块（/config-module/management）；仅需保证执行环境能访问 `POST /api/config-module/backups` 所在 API。

---

## 七、后续可选扩展

- 调度器：对 `run_type=scheduled` 的作业按 `schedule_config` 定时触发 `execute`。
- 流程与作业一对一/多版本：同一流程多次发布是否覆盖同一作业或生成新作业，可在此设计上细化策略。
- 权限：仅管理员可执行、可转为定期等，与现有权限体系对接。
