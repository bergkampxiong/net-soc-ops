# Strix 集成方案

本文档描述如何将 [Strix](https://github.com/usestrix/strix)（开源 AI 安全扫描）集成到 NetOps 工程中，供后续开发与排期参考。

---

## 一、目标与范围

### 1.1 集成目标

- 在 NetOps 平台内**触发 Strix 安全扫描**（支持本地目录、Git 仓库 URL、Web 应用 URL 等目标）。
- **查看扫描结果**（漏洞列表、报告、PoC）并在前端展示。
- 可选：与**流程/作业**结合，在流水线中自动执行扫描；与**配置管理**联动（如对某次配置备份或变更前后做安全检查）。
- 提供 **OpenAPI（LLM）配置界面**，便于在平台内维护 Strix 运行所需的 API 与模型配置，无需改环境变量或服务器。
- 在**原子功能组件库**中新增**渗透测试组件**，与设备连接、配置管理等组件并列，供流程设计器中的「渗透测试」节点引用。

### 1.2 不在此方案内的内容

- 不修改 Strix 上游源码（我们以子模块形式使用，仅通过配置与调用方式集成）。
- LLM 配置（API Key、模型选择）由运维在环境或平台配置中维护，本方案只定义“从哪里读配置”。

---

## 二、Strix 能力与接口简述

### 2.1 能力摘要

- **目标类型**：本地目录、Git 仓库 URL、Web 应用 URL。
- **运行方式**：CLI（`strix --target <目标> [选项]`），依赖 Docker，支持无交互模式（`-n/--non-interactive`）。
- **输出**：结果写入 `strix_runs/<run_name>`，含漏洞报告、PoC 等；退出码非 0 表示发现漏洞。
- **配置**：环境变量 `STRIX_LLM`、`LLM_API_KEY` 等（见 Strix 官方文档）。

### 2.2 与本工程的关系

- **代码位置**：`netops-backend/strix`（Git 子模块，版本随主仓库锁定）。
- **运行环境**：需安装 Docker；Strix 推荐 Python 3.12+，可用其自带 venv/poetry 或系统 Python 安装后调用 CLI。

## 三、集成架构

### 3.1 总体思路

| 层级     | 职责 |
|----------|------|
| **前端** | 扫描任务创建（选择目标类型、填写 URL/路径、可选指令）、任务列表、任务状态与日志、报告查看。 |
| **后端** | 接收扫描请求，校验参数与权限；调用 Strix（CLI 或子进程）；落库任务与结果摘要；提供报告文件或结构化数据接口。 |
| **Strix** | 仅作为“执行引擎”，由后端在指定工作目录或传入参数下启动，不直接对外暴露。 |

### 3.2 调用方式选型

| 方式 | 优点 | 缺点 |
|------|------|------|
| **CLI 子进程** | 与官方使用方式一致，升级 Strix 不影响我方代码；可复用其 Docker 与 venv。 | 需解析 stdout/stderr 或结果目录获取状态与报告。 |
| **Python API** | 可直接调用 `run_cli(args)` 等，便于拿到结构化结果。 | 需将 Strix 加入后端依赖或单独 runner 环境，版本升级可能涉及兼容。 |

**推荐**：**一期采用 CLI 子进程**调用（`subprocess` 或 `asyncio.create_subprocess_exec`），在 `netops-backend/strix` 下激活 Strix 的 venv 后执行 `strix -n --target ...`，结果从 `strix_runs/<run_name>` 读取。后续若需要更细粒度控制或结构化流式输出，再考虑封装 Strix 的 Python 入口。

### 3.3 数据与存储

- **任务表**（新建）：如 `strix_scan_tasks`，字段示例：id、目标类型、目标值（路径/URL）、自定义指令、状态（pending/running/success/failed）、run_name、创建人、创建时间、结束时间、摘要（漏洞数等）。
- **结果**：优先以“结果目录 + 报告文件路径”形式落库或记录；如需结构化展示，可解析 Strix 输出（如 JSON 报告若有）再存表或对象存储。
- **权限**：仅授权用户（如管理员或“安全扫描”角色）可创建任务、查看报告；任务与现有审计/日志可打通。

## 四、技术方案要点

### 4.1 后端

- **路由**：在 `netops-backend` 下新增模块，例如 `routes/strix_integration.py`，挂载于 `/api/config-module/strix` 或 `/api/security/strix`（与现有 config-module、security 路由风格一致）。
- **接口示例**：
  - `POST /scans`：创建扫描任务（body：target_type、target_value、instruction 等）；后端生成 run_name，入表，异步或同步启动 Strix 子进程。
  - `GET /scans`：任务列表（分页、按状态筛选）。
  - `GET /scans/{id}`：任务详情与状态。
  - `GET /scans/{id}/report`：返回报告内容或文件流（从 strix_runs 读取）。
  - `POST /scans/{id}/cancel`：可选，终止正在运行的任务（子进程 kill）。
- **配置**：从环境变量或统一配置表读取 `STRIX_LLM`、`LLM_API_KEY`、Strix 可执行路径、工作目录等；不把 API Key 写死。
- **执行**：工作目录建议固定到某数据目录（如 `data/strix_workspace`），每次任务使用独立子目录或 run_name，避免并发冲突；调用前设置环境变量并激活 Strix venv 后执行 CLI。

### 4.2 OpenAPI（LLM）配置界面

Strix 依赖 LLM 接口（如 OpenAI、Anthropic、Strix Router 等），需配置 `STRIX_LLM`、`LLM_API_KEY` 等。建议在平台内提供**配置界面**，避免直接改服务器环境变量。

| 项 | 说明 |
|----|------|
| **配置内容** | `STRIX_LLM`（如 `openai/gpt-5`）、`LLM_API_KEY`、可选 `LLM_API_BASE`（自建/本地模型）、`PERPLEXITY_API_KEY`（搜索能力）、`STRIX_REASONING_EFFORT` 等，与 [Strix 文档](https://docs.strix.ai) 一致。 |
| **存储** | 后端统一配置表或加密存储（如 `strix_config` 表，或复用现有系统配置表），API Key 需加密或权限控制，仅授权角色可查看/编辑。 |
| **界面位置** | 建议放在**渗透测试组件**页内（见 4.4）的「API 配置」或「运行配置」Tab，与「创建扫描」「任务列表」并列；也可单独作为「安全 → Strix 配置」页。 |
| **接口** | `GET/PUT /api/.../strix/config` 读写当前配置（敏感字段脱敏展示）；执行 Strix 时由后端从该配置注入环境变量，不写死。 |

### 4.3 前端（扫描任务与报告）

- **入口**：在配置管理或“安全”菜单下增加“Strix 安全扫描”页；或统一从**渗透测试组件**入口进入（见 4.4）。
- **功能**：表单（目标类型、URL/路径、可选指令）提交创建任务；任务列表（状态、创建时间、目标）；点击任务查看详情与报告（若为 HTML/文本则内嵌或下载）。
- **权限**：与后端权限一致，仅对具备“扫描”权限的用户展示入口与操作。

### 4.4 渗透测试组件（原子功能组件库）

与现有「设备连接」「配置管理」「配置生成」「数据采集」并列，在**原子功能组件库**中新增**渗透测试组件**，作为 Strix 能力的统一入口，并供流程设计器中的「渗透测试」节点引用。

| 项 | 说明 |
|----|------|
| **路由** | `/rpa/atomic-components/penetration-test`（或 `/rpa/atomic-components/strix`），与现有原子组件一致。 |
| **页面职责** | ① **OpenAPI 配置**：LLM/Strix API 配置表单（对应 4.2）；② **扫描任务**：创建任务、任务列表、报告查看（对应 4.3）；③ 可选：**扫描模板/预设**，保存常用目标与指令，供流程节点或快速创建使用。 |
| **与流程设计器的关系** | 在流程设计器中新增节点类型，如 **渗透测试**（`penetrationTest` / `strixScan`）。节点配置时可选择：使用「默认 Strix 配置」或选择某条「扫描预设」；可填目标、指令等，与设备连接节点选择 SSH 配置类似。代码生成/执行时，调用后端 Strix 扫描接口，传入该节点所选配置与参数。 |
| **后端** | 渗透测试组件所需数据（配置、任务、预设）由 Strix 集成接口统一提供；菜单与路由在「自动化 RPA → 原子功能组件库」下增加「渗透测试」入口。 |

这样，**OpenAPI 配置界面**与**扫描任务/报告**都收敛在**渗透测试组件**内，流程中通过「渗透测试」节点复用同一套能力，与现有原子组件使用方式一致。

### 4.5 与流程/作业的可选联动

- 在流程设计器中增加**渗透测试**节点（见 4.4），参数为 target、instruction、可选扫描预设等，执行时调用后端 Strix 扫描接口。
- 与配置管理联动：如“对某设备某次配置备份的变更内容做安全扫描”，可由流程在备份或变更后触发，target 指向导出的配置文本或临时目录。

## 五、实施阶段建议

| 阶段 | 内容 | 优先级 |
|------|------|--------|
| **Phase 1** | 后端：任务表 + 创建/列表/详情接口；CLI 子进程调用 Strix（-n）；结果目录与 run_name 落库；读取并返回报告文件。 | 高 |
| **Phase 2** | **OpenAPI 配置**：后端配置表/接口（GET/PUT strix config），前端配置表单（LLM 模型、API Key 等）；与执行时环境变量注入打通。 | 高 |
| **Phase 3** | **渗透测试组件**：原子功能组件库新增「渗透测试」页（路由 `/rpa/atomic-components/penetration-test`），集成 OpenAPI 配置 + 扫描任务创建/列表/报告查看；菜单与权限。 | 高 |
| **Phase 4** | **流程设计器**：新增「渗透测试」节点类型，配置面板拉取扫描预设或填目标/指令，代码生成与执行调用 Strix 集成接口。 | 中 |
| **Phase 5** | 可选：扫描预设/模板、与配置管理联动、报告结构化解析与统计、审计日志。 | 低 |

## 六、可视化流程设计器中的渗透测试节点（可实现性）

### 6.1 结论：可以实现，且适合把能力收敛在节点内

与现有「设备连接」「配置备份」「配置下发」节点一致：在流程设计器中**新增「渗透测试」节点类型**，把**目标、指令、扫描模式、可选预设**等全部在**该节点的配置面板**中完成；流程发布后，执行时由**生成的脚本或执行引擎**调用后端 Strix 扫描接口，实现“整条能力嵌入节点”。

### 6.2 与现有流程机制的关系

当前流程执行方式：

- 设计器产出**流程定义**（nodes + edges）。
- **代码生成器**（`CodeGenerator`）根据节点类型生成一段 **Python 脚本**（连接设备 → 配置备份 → 配置下发等）。
- **作业执行**时运行该脚本，完成整条流程。

渗透测试节点可以**沿用同一套机制**：

- 在**代码生成**阶段识别 `penetrationTest` 节点，在生成的 Python 中插入“调用 Strix 扫描 API”的逻辑（创建任务 → 轮询状态 → 可选根据结果失败/继续）。
- 无需改为“按节点解释执行”，只需扩展 `CodeGenerator` 与节点数据契约即可。

### 6.3 扫描目标节点（专门填写目标 URL）

为便于复用目标、并让「扫什么」与「怎么扫」分离，可**新增一类专门填写扫描目标的节点**，渗透测试节点通过**引用该节点**获取 URL，而不是（或除此外）在渗透测试节点内直接填。

| 项 | 说明 |
|----|------|
| **节点类型** | 如 **扫描目标**（`scanTarget`），与「设备连接」「渗透测试」等并列，从左侧拖入画布。 |
| **职责** | **仅负责填写“要扫描的目标”**：目标类型（Web URL / Git 仓库 / 本地路径 / 域名或 IP）、目标值（可多条），以及流程内本地路径的约定（工作区相对路径、或使用本流程配置备份输出目录等）。不对 Strix 的 scan_mode、instruction 等做配置。 |
| **配置面板** | `PDScanTargetPanel`：目标类型下拉、目标值输入（支持「添加目标」多条）、可选「使用本流程配置备份输出目录」等。持久化到节点 `data`：`targetType`、`targets[]`、`useBackupOutput` 等。 |
| **与渗透测试节点的关系** | 渗透测试节点的配置面板中增加**目标来源**：**① 从目标节点获取**：下拉选择当前流程中的某个「扫描目标」节点（按 node id 或展示 label），执行时从该节点的 `data.targets`（及 targetType、useBackupOutput）读取目标并传给 Strix；**② 本节点内填写**：与现有一致，在渗透测试节点内直接填目标类型与目标值。二者二选一，优先「从目标节点获取」时则禁用本节点内的目标输入框。 |
| **执行/代码生成** | 生成脚本时，若渗透测试节点选择了「从目标节点获取」且指向节点 A，则从流程定义中取节点 A 的 `data.targets` 等，作为该次 Strix 调用的 target 入参；若选「本节点内填写」，则用渗透测试节点自身的 `data.targets`。 |

这样，**一个「扫描目标」节点可被多个「渗透测试」节点引用**（例如同一批 URL 用 quick 与 deep 各扫一次），且目标与扫描配置职责清晰。

### 6.4 扫描目标输入（需要扫描的 URL / 目标）

扫描目标来源有两种方式（见 6.3）：

- **方式一：引用「扫描目标」节点**  
  在渗透测试节点配置中选择「从目标节点获取」，并选择画布上的某个「扫描目标」节点；执行时从该节点的 `data` 读取目标类型与目标值（`targets[]`），无需在渗透测试节点内再填 URL。

- **方式二：在渗透测试节点内直接填写**  
  若不使用目标节点，可在渗透测试节点配置中直接填：目标类型、目标值（可多目标）、以及「使用本流程配置备份输出目录」等。  

**「扫描目标」节点或渗透测试节点内填写的目标字段**建议如下：

| 输入项 | 说明 | 示例 |
|--------|------|------|
| **目标类型** | 下拉：**Web 应用 URL**、**Git 仓库 URL**、**本地路径**、**域名或 IP**。 | Web URL / GitHub / 本地路径 / 域名或IP |
| **目标值** | 根据类型填写 URL、路径或域名/IP；支持多个目标时「添加目标」逐条填写，对应 Strix 多个 `-t`。 | `https://myapp.com`、`./app-directory` |
| **流程内“本地路径”** | 若选「本地路径」，执行时约定为作业工作区相对路径或「本流程配置备份输出目录」等，由执行引擎解析为实际路径再传 Strix。 | 工作区 `./target` 或 上一节点输出目录 |

### 6.5 Strix 扫描类型与配置的选择

对应 Strix CLI 的 `--scan-mode`、`--instruction` 等，在节点配置面板中建议：

| 配置项 | 说明 | 可选值 / 示例 |
|--------|------|----------------|
| **扫描模式（scan_mode）** | 必选，控制扫描深度与耗时。 | **quick**：快速，适合 CI/流水线；**standard**：常规；**deep**：深度（默认），全面但耗时更长。 |
| **自定义指令（instruction）** | 可选，文本框。传给 Strix 的 `--instruction`，用于限定范围、提供测试账号、强调某类漏洞等。 | 如：“仅测认证与越权”“使用账号 test/pass 登录后测 API”“重点检查 IDOR 与 XSS” |
| **指令文件（instruction_file）** | 可选。若指令较长，可上传或选择已有文件，执行时传 `--instruction-file`；与 instruction 二选一。 | 流程工作区内路径，如 `./rules-of-engagement.md` |
| **推理强度（reasoning_effort）** | 可选，对应环境变量 `STRIX_REASONING_EFFORT`。可放在节点配置或全局 OpenAPI 配置。 | 如 high / medium（quick 模式常用 medium） |
| **扫描预设** | 可选。从渗透测试组件维护的「扫描预设」中选择一条，自动带出上述部分或全部配置（目标、指令、模式等），减少重复填写。 | 下拉选择预设名称，节点内仍可覆盖部分字段。 |

以上**扫描类型与配置**均在**渗透测试节点的配置面板**内完成；**目标**则可在「扫描目标」节点填写并由渗透测试节点引用，或在渗透测试节点内直接填写。

### 6.6 前端实现要点（汇总）

| 项 | 说明 |
|----|------|
| **扫描目标节点** | 新增节点类型 `scanTarget`（如命名为「扫描目标」），左侧拖拽栏增加入口。节点组件 `PDScanTargetNode`，配置面板 `PDScanTargetPanel`：仅含目标类型、目标值（多目标）、可选「使用本流程配置备份输出」等；持久化 `targetType`、`targets[]`、`useBackupOutput`。 |
| **渗透测试节点** | 在 `nodeTypes` / `nodeConfigs` 中增加 `penetrationTest`，左侧增加「渗透测试」入口。节点组件 `PDPenetrationTestNode`。 |
| **渗透测试配置面板** | `PDPenetrationTestPanel` 内：**① 目标来源**：单选「从目标节点获取」或「本节点内填写」；若选「从目标节点获取」，则下拉选择当前流程中的某个「扫描目标」节点（列出 id 或 label）；若选「本节点内填写」，则展示目标类型 + 目标值输入（同扫描目标节点）。**② 扫描类型与配置**：scan_mode、instruction、instruction_file、可选预设等。持久化 `targetSource`（`"targetNode"` / `"inline"`）、`targetNodeId`（引用时）、`targetType`/`targets[]`（本节点填写时）、`instruction`、`scanMode`、`presetId` 等。 |
| **与设计器联动** | `onNodeClick` 中增加对 `scanTarget`、`penetrationTest` 的分支，分别打开对应配置面板；保存时持久化各节点 `data`。 |

这样，**目标由「扫描目标」节点专门填写，渗透测试节点通过引用该节点获取 URL**；同时保留在渗透测试节点内直接填目标的兼容方式。

### 6.7 后端需要做的

| 项 | 说明 |
|----|------|
| **代码生成** | **渗透测试节点不参与代码生成**（见 6.10）。`CodeGenerator.generate_code()` 仅处理 `deviceConnect`、`configBackup`、`configDeploy`，与当前逻辑一致。 |
| **流程校验** | 后端若有流程校验（如 `CodeGenerator.validate()`），需与前端一致放宽：存在 `penetrationTest` 时，不强制要求设备连接与配置下发/备份。 |
| **作业执行器** | 在 `execute_job` 中：① 按现逻辑生成并执行 Python（仅含设备连接/备份/下发）；② 若流程定义中包含 `penetrationTest` 节点，则解析这些节点及其引用的 `scanTarget`，按流程顺序**调用本系统后端 Strix 集成接口**（如 `POST /api/.../strix/scans`、轮询状态），由后端调 Strix 子模块，不把 Strix 逻辑写进生成脚本。 |
| **执行顺序** | 若流程中既有设备/配置节点又有渗透测试节点，执行顺序约定：先跑完生成脚本（连接→备份→下发），再按 edges 拓扑顺序依次执行渗透测试节点（调 API）；若流程仅有渗透测试节点，则只执行渗透测试 API 调用。 |
| **目标来源** | 执行器解析渗透测试节点时，若配置为「从目标节点获取」且 `targetNodeId` 指向节点 A，则从流程定义中读取节点 A 的 `data.targets`（及 targetType、useBackupOutput），作为该次 Strix API 的 target 入参；否则使用渗透测试节点自身的 `data.targets`。 |

### 6.8 长任务与可选策略

渗透测试可能耗时较长，有两种常见策略，均可实现：

- **同步等待**：生成的代码中轮询扫描状态，直到完成或超时后再执行后续节点；实现简单，但流程会长时间占用执行器。
- **异步提交**：生成的代码只调用“创建扫描”，不轮询；流程继续执行，扫描结果由渗透测试组件/任务列表查看。若需“根据扫描结果决定是否继续”，可后续扩展为“轮询 + 条件分支”或“回调/Webhook 驱动下一节点”。

建议一期采用**同步等待 + 可配置超时**，与现有“执行一段脚本跑到底”的模型一致；后续再视需要增加异步或分支。

### 6.9 小结

- **可否实现**：可以。  
- **目标与扫描分离**：可增加**扫描目标节点**专门填写要扫描的 URL/目标；**渗透测试节点**通过「从目标节点获取」引用该节点得到 URL，也可选择在本节点内直接填写目标。  
- **实现量级**：前端新增 **扫描目标**（`scanTarget`）与**渗透测试**（`penetrationTest`）两种节点及对应配置面板；后端按下方 6.10 的验证与执行模型配合，无需破坏现有作业执行框架。

### 6.10 流程保存、验证、发布与执行（您提出的四点）

针对「能否保存流程」「渗透测试是否参与代码生成」「发布是否进作业执行控制」「能否被作业执行控制执行」的结论与建议如下。

**1. 这样能保存流程吗？流程不是有验证节点的吗？**

- **现状**：当前流程校验要求「必须包含开始、结束、至少一个设备连接、至少一个配置下发或配置备份」。若画布上只有「扫描目标 + 渗透测试」而没有设备连接/配置节点，**会校验不通过，无法保存**。
- **建议**：**放宽校验规则**，使含渗透测试的流程也能保存与发布：
  - 规则可改为：**必须包含开始 + 结束**；且 **至少满足下列之一**：  
    （a）至少一个设备连接 +（至少一个配置下发或配置备份）  
    或（b）至少一个渗透测试节点（可选带扫描目标节点）。  
  - 即：纯「扫描目标 + 渗透测试」的流程允许保存；同时保留「设备连接 + 配置下发/备份」或「设备连接 + 配置 + 渗透测试」混合流程。
- **落地**：前端 `validateProcess()` 与后端 `CodeGenerator.validate()` 同步放宽（例如：若存在 `penetrationTest` 节点，则不再强制要求 deviceConnect 与 configDeploy/configBackup）。

**2. 渗透测试是不是就不要代码生成了？他不属于 netops 的线程执行（子模块）。**

- **建议**：**渗透测试节点不参与“生成 Python 脚本”**。Strix 以子模块/独立进程方式运行，不放在 NetOps 生成的同一条 Python 线程里执行更清晰。
- **执行方式**：  
  - **代码生成**：仅针对「设备连接、配置备份、配置下发」等现有节点生成 Python，与现在一致。  
  - **渗透测试节点**：不在生成代码里出现，改为在**作业执行器**里单独处理：执行完生成的脚本后（或按流程顺序），根据流程定义中的 `penetrationTest`（及引用的 `scanTarget`）节点，**调用本系统后端 Strix 集成接口**（如 `POST /api/.../strix/scans`），由后端负责调 Strix CLI/子进程，作业侧只负责传参与轮询状态。
- 这样「谁在跑 Strix」明确：NetOps 后端服务调 Strix，作业执行器只发起 HTTP 请求，不把 Strix 逻辑写进生成的脚本。

**3. 流程发布一致吗？流程管理发布之后会进入作业执行控制吗？**

- **一致**。当前逻辑就是：流程管理里**发布**某流程定义后，会创建或更新一条 **Job**（`process_definition_id` 指向该流程），该 Job 会出现在**作业执行控制**列表中。  
- 因此，只要流程（含扫描目标 + 渗透测试）能通过上述放宽后的校验并保存，发布后就会在作业执行控制里看到对应作业，行为与现有「设备连接 + 配置下发/备份」流程一致。

**4. 也可以被作业执行控制执行吗？**

- **可以**。作业执行控制里点击「执行」该作业时：
  - 若有**设备连接/配置备份/配置下发**节点：与现在一样，生成 Python 并 subprocess 执行。
  - 若有**渗透测试**节点：按 6.10 第 2 点，**不**把这些节点写进生成代码，而是在执行器中**按流程定义解析渗透测试（及扫描目标）节点**，向后端发起创建扫描、轮询状态的请求；执行记录、日志仍挂在同一次作业执行下。
- 即：**同一份流程发布出来的作业，既可以包含“设备+配置”的脚本执行，也可以包含“渗透测试”的 API 调用**，都在作业执行控制里一次执行完成。

### 6.11 渗透测试报告与作业监控

**问题**：当前「作业监控与报告」里的**作业详情**面向的是脚本执行结果——展示执行状态、`result`（如 returncode）、`logs`（stdout/stderr）、错误信息等，**无法满足 Strix 的展示需求**（漏洞列表、严重程度分布、扫描摘要、报告文件下载等）。若把 Strix 的完整结果塞进作业详情的日志或 result 里，既不便于安全侧查看，也不利于后续按扫描维度筛选与审计。

**建议**：**单独为 Strix 做「渗透测试报告」模块**，放在「任务作业管理」目录下，与「作业执行控制」「作业监控与报告」并列；作业详情仅做轻量联动，不承载完整报告。

| 维度 | 说明 |
|------|------|
| **菜单位置** | 在「任务作业管理」下新增菜单项：**渗透测试报告**（或「扫描报告」），路由如 `/rpa/task-job-management/penetration-reports`。 |
| **列表页** | 列表按「扫描」维度展示：关联作业执行 ID、作业名称、扫描目标、扫描时间、状态（运行中/完成/失败）、严重程度汇总（如 高/中/低 数量）等；支持按时间范围、作业、目标、状态筛选。数据来源：执行器在调用 Strix 创建扫描时，将 `job_execution_id`、`scan_id`（Strix 返回）等落库或由后端 Strix 集成层统一记录，供前端拉取。 |
| **详情页** | 单次扫描的完整报告：漏洞列表（含名称、严重程度、状态、描述）、可选的严重程度分布图、扫描时长、目标信息；提供「下载报告」（Strix 原生 HTML/PDF 等，若后端已落盘或可实时生成）。 |
| **与作业详情的联动** | 在**作业详情**的「执行历史」中，当某条执行记录包含渗透测试节点时，在该行增加「渗透测试」列或操作按钮「查看渗透测试报告」，跳转到渗透测试报告列表并过滤 `job_execution_id = 当前执行 ID`，或直接打开该次执行对应的扫描报告详情（若一次执行对应多次扫描，可进入列表页再选）。不在作业详情里复刻完整漏洞表，避免页面过重且与「渗透测试报告」重复维护。 |

这样**职责清晰**：作业监控与报告继续负责「作业/执行维度」的日志与状态；渗透测试报告负责「扫描维度」的漏洞与报告，安全人员可从任务作业管理进入渗透测试报告做专项查看与导出，同时从作业详情能快速跳转到本次执行的渗透测试报告。

## 七、依赖与风险

### 7.1 依赖

- **Docker**：Strix 依赖 Docker 运行沙箱，需保证执行 Strix 的宿主机可访问 Docker（与当前安装脚本一致）。
- **Strix 版本**：以当前子模块指向的版本为准；升级 Strix 时需回归 CLI 参数与输出格式（见 [INSTALL.md 8.4 / 8.5](../INSTALL.md)）。
- **LLM**：需事先配置可用 LLM API Key，否则扫描会失败。

### 7.2 风险与缓解

- **长时间任务**：扫描可能耗时较长，建议任务异步执行、超时可配置、支持取消。
- **资源**：单机并发扫描数需限制，避免同时拉起过多 Docker 容器。
- **敏感数据**：报告可能含 PoC 或敏感信息，仅授权用户可访问，且不对外暴露 Strix 工作目录。
