# 原子功能组件库与可视化流程设计器 — 功能梳理与重构开发计划

## 一、当前原子功能组件说明

### 1.1 已有明确实现的三个组件

| 组件 | 路由 | 作用 | 后端/数据 |
|------|------|------|-----------|
| **设备连接组件** | `/rpa/atomic-components/device-connections` | 管理 SSH 连接模板（名称、设备类型、凭证、端口、Netmiko 参数），以及连接池监控（Redis/网络设备）、清空异常连接；连接池统计与 `device_connection_manager` 已打通，页面上有与 SSH 连接配置的关联说明。 | `DeviceConnection` 表、`/api/device/connections`、Redis `device_connection_stats` 等 |
| **配置管理组件** | `/rpa/atomic-components/config-management` | 管理配置模板文件：支持 Jinja2、TextFSM、作业(job) 三种类型，CRUD、版本、导入导出。 | `/api/config/files`、`ConfigManagementService`、config 相关表 |
| **配置生成组件** | `/rpa/atomic-components/config-generator` | 从配置库中选取 **Jinja2 模板**，填写 JSON 变量，前端用 Nunjucks 渲染生成配置内容，可保存为 **job 类型** 配置到配置库。 | `/api/config-generator/templates`（jinja2）、保存走 `/api/config/files` |

### 1.2 数据采集组件的现状与建议定位

| 组件 | 路由 | 当前实现 | 建议作用（规划） |
|------|------|----------|------------------|
| **数据采集组件** | `/rpa/atomic-components/data-collection` | 仅占位页，`Empty` 提示「数据采集组件暂无内容」。作业类型中有 `data_collection` 选项。 | **建议**：管理「采集任务模板」或「采集配置」（如：从设备执行哪些命令、解析方式 TextFSM、落库/上报策略），为流程中的「数据采集」节点或作业提供可复用配置。与配置管理、设备连接配合。 |

**说明**：安全审计组件、告警与报告组件已从产品中移除（无对应前端页面与菜单）。

## 二、原子功能组件与流程设计器节点的对应关系

### 2.1 流程设计器中的节点类型（当前）

- **前端**（`pd-flow-designer.tsx` / `pd-node-panel.tsx`）使用：`deviceConnect`、`configDeploy`、`commandExecute`、`configBackup`、`statusCheck`，以及 `start`、`end`、`task`、`condition`（含 `pd_*` 枚举在类型定义中）。
- **后端**（`app/process_designer/code_generator.py`）**仅使用**：`deviceConnect`、`configDeploy`；且校验「必须至少一个设备连接节点、至少一个配置下发节点」。

### 2.2 组件 ↔ 节点 对应关系与缺口

| 原子功能组件 | 设计器节点 | 关联情况 | 说明 |
|--------------|------------|----------|------|
| 设备连接组件 | **设备连接**（deviceConnect） | ✅ 强关联 | 设计器「设备连接」配置面板拉取 SSH 配置列表（`getSSHConfigs()`）、设备组；代码生成时用 `sshConfig` + `selectedDevices` 生成连接与连接池调用。 |
| 配置管理组件 | **配置下发**（configDeploy） | ⚠️ 间接 | 配置管理存 jinja2/textfsm/job；设计器「配置下发」从 **job-templates**（`/api/config-generator/job-templates`，即 template_type=job 的配置）选模板带出内容，**未直接**从「配置管理」按 id 选模板，也未用 jinja2 渲染。 |
| 配置生成组件 | 配置下发 | ⚠️ 间接 | 配置生成产出 job 类型配置入库后，会出现在 job-templates 列表中，供设计器配置下发选用。无单独节点。 |
| 数据采集组件 | （无对应节点） | ❌ 无 | 设计器无「数据采集」节点；作业类型有 data_collection，但无节点与组件绑定。 |
| — | **命令执行**（commandExecute） | ❌ 未参与代码生成 | 仅在设计器中有配置（命令、超时），后端代码生成未使用。 |
| — | **配置备份**（configBackup） | ❌ 未参与代码生成 | 仅在设计器中有配置（路径、文件名），后端代码生成未使用。 |
| — | **状态检查**（statusCheck） | ❌ 未参与代码生成 | 仅在设计器中有配置（检查项），后端代码生成未使用。 |

### 2.3 小结：关系模糊与不合理之处

1. **命名/类型不统一**：设计器用 `deviceConnect`/`configDeploy` 等 camelCase，类型定义用 `pd_device_connect` 等，存在两套命名，易混淆。
2. **配置下发与配置库脱节**：配置下发节点只拉「job-templates」，不按配置管理里的 id/版本选择，也不支持在流程中传参渲染 jinja2。
3. **数据采集为占位**：数据采集组件仅有占位页，无能力、无对应节点，与作业类型 data_collection 未打通。
4. **多节点未落地**：命令执行、配置备份、状态检查在设计器可配置，但未进入代码生成与执行，流程无法真正执行这些步骤。
5. **配置生成与配置管理概念重叠**：配置管理已包含 jinja2/job；配置生成本质是「用 jinja2 渲染并另存为 job」，未在菜单或文案上区分「模板库」与「生成器」的边界。

## 三、当前功能设计中的不合理之处

1. **原子组件与节点不对齐**：原子功能组件库现有四类（设备连接、配置管理、配置生成、数据采集），其中三个已实现、一个为占位；五类网络自动化节点仅两类参与代码生成，其余悬空。
2. **模板数据源分散**：配置管理（config/files）、配置生成（config-generator/templates）、设计器（config-generator/job-templates）三条接口，概念上都是「配置/模板」，但未统一为「配置库 + 类型 + 用途」的清晰模型。
3. **设备连接与连接池**：~~设备连接组件内「连接池管理」同时展示 Redis 与网络设备连接池，与「SSH 连接配置」的关联在界面上不直观；且连接池统计与 `device_connection_manager` 的写入若未打通，数据可能不准确。~~ **已优化**：后端 stats/metrics/cleanup 已按 `pool_type` 区分，网络设备池统一读取 `device_connection_stats`（与 `device_connection_manager` 写入一致），并兼容 Redis 返回的 str/bytes key；前端默认展示「网络设备连接池」并传递 `pool_type`，连接池管理页已增加「与 SSH 连接配置的关系」说明。
4. **作业类型与组件/节点脱节**：作业有「数据采集」等类型，但无对应原子组件能力、无设计器节点，无法形成闭环。
5. **代码生成与执行假设**：CodeGenerator 假设 `DeviceConnectionManager.get_connection(device_params)` 等接口形态，与当前后端按 `connection_id + host` 从池取连接的实现可能不一致，需核对。

## 四、重构与开发计划（建议）

### 阶段一：统一概念与映射（文档 + 前端收敛）

| 序号 | 内容 | 说明 |
|------|------|------|
| 1.1 | **明确四类原子组件的职责** | 在产品/开发文档中固定：设备连接、配置管理、配置生成、数据采集的边界与流程/作业的衔接方式。 |
| 1.2 | **统一设计器节点类型命名** | 前端统一使用一套类型（建议与后端约定一致，如 `deviceConnect` / `configDeploy`），类型定义与节点配置、代码生成使用同一枚举或常量。 |
| 1.3 | **建立「组件–节点」映射表** | 在代码或文档中维护：哪些节点必须依赖哪类原子组件（如设备连接节点 ↔ 设备连接组件中的 SSH 配置），配置下发节点 ↔ 配置管理/配置生成产出的 job 模板。 |

### 阶段二：打通配置与流程

| 序号 | 内容 | 说明 |
|------|------|------|
| 2.1 | **配置下发节点对接配置库** | 配置下发节点支持从「配置管理」按 id（及可选版本）选择模板；若选 jinja2，支持在节点上填变量或从上下文传参，由后端渲染后下发。 |
| 2.2 | **统一模板/配置 API 语义** | 将「config/files」「config-generator/templates」「config-generator/job-templates」收敛为：同一配置库，按 template_type（jinja2/textfsm/job）与用途（生成器用/设计器用）筛选，避免多套「模板」概念。 |
| 2.3 | **配置生成组件定位** | 明确为「基于 Jinja2 模板 + 变量生成配置并保存到配置库」的快捷入口，与配置管理「模板 CRUD」区分；在配置下发节点中可选用配置生成产出的 job。 |

### 阶段三：补齐节点与代码生成

| 序号 | 内容 | 说明 |
|------|------|------|
| 3.1 | **命令执行节点落地** | 后端代码生成与执行引擎解析 `commandExecute` 节点，在已有连接上执行命令列表，输出可写入上下文或日志。 |
| 3.2 | **配置备份节点落地** | 后端解析 `configBackup` 节点，在已有连接上执行备份命令（或按设备类型适配），落盘或上传到指定路径/存储。 |
| 3.3 | **状态检查节点落地** | 后端解析 `statusCheck` 节点，按检查项（如接口状态、BGP/OSPF）执行对应命令并解析结果，成功/失败驱动分支或告警。 |
| 3.4 | **校验与代码生成解耦** | 将「必须设备连接+配置下发」改为可配置规则或模板：如「批量配置下发」模板必须二者，「仅巡检」模板可仅设备连接+状态检查等。 |

### 阶段四：数据采集组件规划

| 序号 | 内容 | 说明 |
|------|------|------|
| 4.1 | **数据采集组件** | 定义「采集模板」或「采集任务」：命令、解析方式（如 TextFSM）、目标（CMDB/监控/报表）。设计器新增「数据采集」节点，绑定采集模板；作业类型 data_collection 与节点/模板关联。 |

### 阶段五：连接池与设备连接一致性

| 序号 | 内容 | 说明 | 状态 |
|------|------|------|------|
| 5.1 | **连接池统计与真实池一致** | 确保「网络设备连接池」的统计数据来自 `device_connection_manager` 的实际使用（或 Redis 中 `device_connection_stats`），避免界面与真实池不一致。 | **已完成**：后端 `get_pool_status`/`get_pool_metrics`/`cleanup` 已支持 `pool_type`，device 时统一读/写 `device_connection_stats` 并兼容 str/bytes key；前端传 `pool_type`，默认「网络设备连接池」；连接池页已增加与 SSH 连接配置的关联说明。 |
| 5.2 | **设备连接与设计器/执行一致** | 代码生成与执行时使用「SSH 配置 id + host」从池取连接，与设备连接组件中维护的 `DeviceConnection` 一致；文档中明确「设计器选用的是原子组件中的哪条 SSH 配置」。 | 待核对 |

## 五、文档与代码层面的建议

- **在仓库中保留本文档**（如 `docs/原子功能组件与流程设计器_重构开发计划.md`），随实现更新「当前实现」与「对应关系」两节。
- **在原子组件各页面**增加简短说明：本组件的作用、与流程设计器/作业的关联（可链到本重构计划或产品说明）。
- **流程设计器**：在「配置下发」等节点配置面板中，增加「从配置库选择」的明确入口，并注明数据来源（配置管理 / 配置生成产出的 job）。
- **后端**：代码生成与执行引擎与 `DeviceConnectionManager`、配置库 API 的对接处做一次对照检查，确保参数与返回值一致，并在本计划中记录结论。

## 六、总结

- **原子功能组件**：设备连接、配置管理、配置生成已实现并有明确作用；数据采集为占位，需在本文规划下赋予职责并与流程/作业打通。安全审计、告警与报告已从产品中移除。
- **组件与设计器节点**：仅设备连接 ↔ 设备连接节点、配置下发 ↔ job 模板 有实质关联；配置管理与配置生成通过「job 模板」间接关联；命令执行/配置备份/状态检查节点未参与代码生成；数据采集组件暂无对应节点。
- **重构重点**：统一节点命名与数据源、配置下发与配置库/jinja2 打通、补齐命令执行/配置备份/状态检查的生成与执行、明确并实现数据采集组件、保证连接池与设备连接在设计与运行期一致。
- **已完成的优化**：连接池统计已与 `device_connection_manager` 打通（后端按 `pool_type` 读 `device_connection_stats`，兼容 key 类型）；连接池管理页已默认展示网络设备连接池并增加与「SSH 连接配置」的关联说明。
