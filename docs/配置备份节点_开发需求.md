# 可视化流程设计器 — 配置备份节点 开发需求

> 基于已实现的**配置管理模块**（设备配置备份库、配置摘要、配置管理页面），重新构建流程设计器中的「配置备份」节点，使流程执行时从设备拉取配置并写入配置管理模块，供后续在「配置管理」中查看、对比、检索。  
> 本文档仅描述开发需求与实现要点，不修改其它功能代码与数据库（配置管理模块已就绪）。

---

## 一、目标与范围

### 1.1 目标

- **配置备份节点**：在流程中表示「从网络设备拉取当前配置并保存」这一步骤。
- **执行语义**：对流程中已连接的目标设备（来自前置「设备连接」节点），逐台执行备份命令（如 `show running-config`），将输出全文写入**配置管理模块**的备份库（`config_module_backups`），并可在「配置管理」菜单下查看、按设备/时间检索、版本对比。
- **与配置管理模块的边界**：作业/调度由原子工作流或任务作业负责；本节点只负责「单次流程执行时的备份动作」及写入 `POST /api/config-module/backups`。

### 1.2 关键问题说明（FAQ）

**Q1：备份的配置是保存到数据库还是保存到文件？**  
- **答：保存到数据库。** 配置全文写入配置管理模块的表 `config_module_backups`（字段 `content`），通过接口 `POST /api/config-module/backups` 提交。不在本需求中落盘为本地文件；若需导出文件，用户在「配置管理」页自行导出即可。

**Q2：如何区分 Cisco、华为、H3C、PA、锐捷、飞塔等厂商并执行对应备份命令？**  
- **答：通过设备连接节点所选「SSH 配置」中的 `device_type` 区分。** 流程执行时，配置备份节点使用的设备列表来自其关联的「设备连接」节点；该节点保存了 `sshConfig`（含 `device_type`）。每个设备共用该连接节点的 `device_type`。代码生成/执行端维护「device_type → 默认备份命令」映射表（见 3.1 节），按 `device_type` 选用 Cisco/华为/H3C/PA/锐捷/飞塔等对应命令；若节点配置了「备份命令（覆盖）」则优先使用覆盖命令。

**Q3：代码生成会生成对应设备厂商的备份命令吗？**  
- **答：会。** 生成代码（或服务端执行逻辑）中必须包含「device_type → 默认备份命令」的映射；对每台设备根据其 `device_type`（来自设备连接节点的 sshConfig）选择命令后再执行。若配置备份节点填写了「备份命令（覆盖）」，则所有设备统一使用该覆盖命令，不再按厂商区分。

**Q4：写入备份时的 device_id、device_name、device_host、content、source、remark 这些字段从哪里来？**  
- **答：** 见下文 **3.3 写入配置管理模块的字段映射（数据来源详解）**。

### 1.3 范围

- **前端**：流程设计器内「配置备份」节点的配置面板、节点数据结构。
- **后端**：流程代码生成或执行逻辑中，增加「配置备份」步骤：拉取配置 + 调用配置管理模块写入接口。
- **不修改**：配置管理模块现有 API、表结构、配置摘要/配置管理/合规等页面；设备连接节点、配置下发节点等其它节点逻辑。

## 二、当前配置备份节点现状

### 2.1 前端

| 位置 | 现状 |
|------|------|
| **节点** | `pd-config-backup-node.tsx`：仅展示「配置备份节点」标题与图标，无业务配置展示。 |
| **配置面板** | `pd-config-backup-panel.tsx`：字段为「备份名称」「备份描述」「备份路径」，无设备来源、无与配置管理模块的关联。 |
| **类型定义** | `automation.ts` 中 `ConfigBackupNode` 含 name、description、backupPath、backupType、compress、retentionDays 等，与「写入配置管理备份库」无对应关系。 |

### 2.2 后端

| 位置 | 现状 |
|------|------|
| **代码生成** | `app/process_designer/code_generator.py`：仅处理 `deviceConnect` 与 `configDeploy`，**未处理 `configBackup`**。 |
| **校验** | 当前校验要求「至少一个设备连接节点、至少一个配置下发节点」，未考虑「仅备份不下发」的流程。 |
| **执行** | 流程执行时无「拉取配置并写入配置管理模块」的逻辑。 |

### 2.3 配置管理模块（已就绪）

- **写入接口**：`POST /api/config-module/backups`  
  - 请求体：`device_id`（必填）, `device_name?`, `device_host?`, `content`（必填）, `source?`, `remark?`  
  - 建议：`source=workflow`，`remark` 可带流程名称或执行 ID。
- **存储**：表 `config_module_backups`，支持按设备、时间、关键词在「配置管理」页检索与版本历史、diff。

## 三、功能需求

### 3.1 节点配置项（前端面板）

| 配置项 | 类型 | 必填 | 说明 |
|--------|------|------|------|
| **设备来源** | 枚举 | 是 | 「使用流程中设备连接节点的设备」：从本流程中设备连接节点取设备列表（SSH 配置 + 设备 IP/主机）。若流程中存在多个设备连接节点，可约定取第一个或由用户选择。 |
| **备注** | 单行文本 | 否 | 写入备份时的 `remark`，如「日常备份」「变更前快照」，便于在配置管理页筛选。 |
| **备份命令（可选）** | 单行文本 | 否 | 默认按设备类型使用内置命令（见下表）；可覆盖为自定义命令。 |

**设备类型与默认备份命令（与 Netmiko 一致，代码生成/执行时按 device_type 选用）**：

| 设备类型（device_type） | 默认备份命令 |
|--------------------------|--------------|
| cisco_ios / cisco_xe / cisco_nxos 等 Cisco 系 | `show running-config` |
| huawei / huawei_vrp 等 华为 | `display current-configuration` |
| h3c / hp_comware 等 H3C | `display current-configuration` |
| paloalto_panos 等 PA | `show config running`（以 PA 官方为准，可调整） |
| ruijie / ruijie_os 等 锐捷 | `show running-config`（或 `show run`，以实际设备为准） |
| fortinet 飞塔 | `get system running-config`（以飞塔官方为准，可调整） |
| 其它 | `show running-config`（可后续按需扩展） |

- **不再使用**：备份名称、备份描述、备份路径、backupType、compress、retentionDays（这些属于「本地/文件备份」语义，与配置管理模块的「写入备份库」语义不同；若需保留字段可仅作展示或废弃）。

### 3.2 节点与流程的关系

- **前置依赖**：配置备份节点应至少有一个**设备连接节点**在前（通过连线或流程顺序），以便获得「设备列表 + SSH 连接信息」。
- **执行顺序**：流程执行时建议顺序为：开始 → 设备连接（建立连接池/连接）→ 配置备份（按设备拉取配置并写入配置管理模块）→ （可选）配置下发等 → 结束。
- **多设备**：对设备连接节点选中的每台设备执行一次「拉取配置 → POST /backups」，实现批量备份。

### 3.3 写入配置管理模块的字段映射（数据来源详解）

以下字段均在**执行时**由代码生成/执行引擎从流程定义与运行时结果中取得，并调用 `POST /api/config-module/backups` 传入：

| 配置管理模块字段 | 取值来源（如何找到） |
|------------------|------------------------|
| **device_id** | 来自设备连接节点的 **selectedDevices** 中当前设备的 IP（或主机名）。建议直接使用该值作为设备唯一标识；若需区分端口可拼接为 `host:port`。即：遍历设备连接节点的 `data.selectedDevices`，每个元素即为该次备份的 device_id。 |
| **device_name** | 可选。设备连接节点若保存了设备名称（如设备组中的设备名），可从该处取；否则可执行 `show hostname` 等命令解析，或留空。当前设备连接节点主要保存的是 **selectedDevices**（IP 列表）和 **sshConfig**，无单独“设备名”列表时可为空。 |
| **device_host** | 与 device_id 相同，取当前设备的 IP 或主机名，即 **selectedDevices[i]** 或设备连接节点里该设备对应的 host。 |
| **content** | 当前设备上执行**备份命令**后的完整标准输出（全文）。即对已建立的 Netmiko 连接执行 `send_command(backup_command)` 的返回值。 |
| **source** | 固定写死为字符串 `workflow`，表示来自流程执行。 |
| **remark** | 配置备份节点配置项中的「**备注**」字段（用户填写）；可在此基础上前缀或后缀流程名称、执行 ID，便于在配置管理页筛选。 |

## 四、实现要点

### 4.1 前端

1. **配置备份面板（pd-config-backup-panel.tsx）**
   - 增加「设备来源」：单选「使用流程中设备连接节点的设备」；若有多个设备连接节点，可下拉选择「使用哪个设备连接节点的设备」（按节点 id 或 label）。
   - 增加「备注」输入框。
   - 可选：增加「备份命令（覆盖）」输入框，为空则用默认命令。
   - 移除或隐藏与「备份路径、备份类型、压缩、保留天数」相关的表单项（或保留字段但不参与后端逻辑）。

2. **节点数据结构（ConfigBackupNode / data）**
   - 建议字段：`useDeviceFromNodeId?: string`（关联的设备连接节点 id）、`remark?: string`、`backupCommand?: string`（可选覆盖命令）。
   - 保存/加载流程时持久化上述字段。

3. **校验**
   - 若存在配置备份节点，流程中应存在至少一个设备连接节点，且配置备份节点的「设备来源」指向有效节点。

### 4.2 后端（代码生成或执行引擎）

1. **代码生成（code_generator.py）**
   - 解析流程中的 `configBackup` 节点，获取其关联的 `deviceConnect` 节点及设备列表（含 `data.sshConfig.device_type`、`data.selectedDevices`）。
   - 在生成的 Python 中增加步骤（示例逻辑）：
     - 在 `_connect_devices` 之后，对每台设备：根据 **device_type**（来自 sshConfig）查「device_type → 默认备份命令」映射表得到备份命令，若配置备份节点配置了「备份命令（覆盖）」则用覆盖命令；执行 `connection.send_command(backup_command)` 获取配置全文；
     - 调用内网 API `POST /api/config-module/backups`，传入 device_id=host、device_host=host、content=命令输出、source=workflow、remark=节点备注（见 3.3 节字段来源）。
   - **必须生成按厂商区分的备份命令**：在生成代码内维护映射表（如字典），覆盖 3.1 节所列 Cisco/华为/H3C/PA/锐捷/飞塔等 device_type，确保不同厂商设备执行正确命令。
   - 若流程中无配置下发节点、仅有配置备份节点，校验逻辑应允许「仅设备连接 + 配置备份」的流程（不再强制要求存在 configDeploy 节点）。

2. **执行环境**
   - 执行生成的 Python 时，需能访问到后端 API（同机或配置 BASE_URL）；或改为由后端「流程执行服务」直接解析流程、执行备份逻辑并调用 `POST /backups`，而不生成 Python。具体采用「生成代码」还是「服务端解释执行」由现有架构决定，需求上均需实现「拉取配置 + 写入配置管理模块」。

3. **备份命令与设备类型**
   - 建议在代码生成或执行逻辑中维护「device_type → 默认 backup_command」映射表，与 3.1 节一致；节点配置的「备份命令（覆盖）」优先使用。

### 4.3 与设备连接节点的数据衔接

- 设备连接节点保存的数据中通常包含：`sshConfig`（含 device_type、port、username、password 等）、`selectedDevices`（设备 IP 列表）或设备组信息。
- 配置备份节点需能通过「设备来源」拿到同一份设备列表及 SSH 参数，用于建立连接并执行备份命令；若当前保存结构中没有「节点 id」关联，可通过「顺序」或「选择设备连接节点」下拉框指定取哪个节点的设备列表。

## 五、验收标准

- [ ] 流程设计器中配置备份节点可配置「设备来源」「备注」及可选「备份命令」。
- [ ] 保存流程后，再次打开流程，配置备份节点配置可正确回显。
- [ ] 流程执行（或生成代码执行）后，对设备连接节点选中的每台设备执行备份命令，并将结果写入 `config_module_backups`（source=workflow）。
- [ ] 在「配置管理」菜单下可按设备、时间查到本次备份记录，且内容与设备实际上运行的配置一致。
- [ ] 不修改配置管理模块现有 API、表结构及其它节点（设备连接、配置下发）的既有行为；仅扩展配置备份节点与代码生成/执行逻辑。

## 六、与配置管理模块设计计划对照

| 配置管理模块设计计划 | 本需求 |
|----------------------|--------|
| 流程设计器 - 配置备份节点需落地：执行时从设备拉取配置并写入本模块的配置库 | 本需求即实现该落地：配置备份节点重构建 + 执行时调用 POST /api/config-module/backups |
| 作业不在本模块实现，由原子工作流实现 | 本需求仅实现「单次流程执行」中的备份；定时/批量由工作流或任务作业编排多次执行该流程 |

## 七、附录：配置管理模块备份接口速查

- **URL**：`POST /api/config-module/backups`
- **Body**：  
  `{ "device_id": "必填", "device_name": "可选", "device_host": "可选", "content": "必填-配置全文", "source": "可选-建议workflow", "remark": "可选" }`
- **响应**：201 + 备份对象（含 id、created_at 等）。
