# 渗透测试功能优化说明

本文档基于对 Strix 工程与当前 net-soc-ops 集成代码的阅读，说明两项优化：**非静态测试时提供系统账号密码以提升测试效果**（含在流程设计器中的落点与前端 UI 提示）、**Strix 测试报告的读取与人类友好统一报告方案**（语言与层次性、业界格式结合 Strix 输出）。**文内已纳入《统一渗透测试报告实现方案》的完整内容，便于整体作为 PRD 编写依据。**

---

## 一、非静态测试时提供账号密码是否提高测试结果？在哪个节点添加？

### 1.1 结论与建议

- **会提高。** 在非静态（动态/黑盒或灰盒）测试场景下，提供**测试用系统账号与密码**可以让 Strix 进行**已认证扫描**，显著提高测试深度与结果质量：
  - 覆盖需登录后才能访问的接口与页面（越权、敏感数据泄露、业务逻辑漏洞等）；
  - 减少“未授权无法测试”的盲区；
  - 更贴近业界渗透测试中“授权测试账号”的实践。
- **在可视化流程设计器中的落点**：应在 **「渗透测试」节点** 的配置中增加账号密码输入，而不是在「扫描目标」节点。理由：扫描目标节点仅描述“测谁”（URL/仓库/路径与类型），而“用什么身份测”属于渗透测试执行参数，与自定义指令同属渗透测试节点更合理。

### 1.2 Strix 侧依据

- Strix 通过 **`--instruction`**（对应内部 `user_instructions`）接收自定义说明，并拼接到 Agent 的任务描述中（`strix_agent.py`: “Special instructions: {user_instructions}”）。
- 当前工程在创建扫描时已将「渗透测试」节点中的 `instruction` 传给 Strix（`job.py` / `strix_integration.py` 的 `instruction` 字段）。因此，**把测试账号密码以约定格式写入 instruction，或由前端单独字段拼入 instruction，Strix 即可在测试中使用**（例如登录后再测已认证接口）。

### 1.3 前端 UI 落点与提示建议

在 **流程设计器 → 渗透测试节点配置面板**（`pd-penetration-test-panel.tsx`）中：

- **新增表单项（可选）：**
  - **测试账号**（如 `username` / 登录名）
  - **测试密码**（`password`，建议使用 `Input.Password`，并在提交时与 instruction 一起下发）
- **放置位置建议：** 放在「自定义指令（可选）」上方或下方，与「扫描模式」同区域，便于用户理解“执行参数”为一组。
- **UI 提示文案建议：**
  - 在「测试账号/测试密码」区域上方或旁加 **说明/Alert**：
    - **“非静态测试时，可填写被测系统的测试账号与密码，以便进行已认证扫描，提高漏洞发现率；仅用于您已授权的测试环境。”**
  - 若目标来源为「仅静态（代码审计）」或 `staticOnly === true`，可将账号/密码表单项 **禁用或折叠**，并提示：“仅静态扫描不需要系统账号密码。”

这样，用户仅在**非静态测试**时看到并填写账号密码，且明确知道用途与授权范围。

### 1.4 后端配合

- 作业执行时（如 `job.py` 中处理渗透测试节点）：若节点数据中带有 `testUsername` / `testPassword`（或与前端约定好的字段名），则将其格式化为一段说明（例如：“使用以下测试账号进行已认证扫描：用户名 {username}，密码 {password}。请先登录后再进行需认证的接口与页面测试。”），**拼接到现有 `instruction` 前部**，再传给 Strix 的 `POST /config-module/strix/scans` 的 `instruction` 字段。注意：密码需通过安全方式传递（如仅内存拼接、不落库、不写日志），避免泄露。

## 二、当前工程能否读取 Strix 生成的测试报告？报告样式与人类友好性

### 2.1 能否读取

- **能读取。** 当前逻辑如下：
  - 扫描完成后，Strix 在 **run 目录**（如 `data/strix_workspace/<workspace>/strix_runs/<run_name>/`）下生成：
    - **`penetration_test_report.md`**：总报告（Executive Summary、Methodology、Technical Analysis、Recommendations）；
    - **`vulnerabilities/`**：每个漏洞一个 `vuln-xxxx.md`；
    - **`vulnerabilities.csv`**：漏洞列表索引。
  - 本工程通过 `task.report_path` 保存该 run 目录（或子目录），并在 **`GET /config-module/strix/scans/{task_id}/report`** 中根据 `report_path` 查找文件。
  - **当前限制：** 报告接口**仅查找并返回 `.html` 文件**；而 Strix **默认只生成 Markdown（.md）**，不生成 HTML。因此，在典型 Strix 输出下，前端“下载报告”可能得到“No HTML file found”或类似提示，**即：工程能“读到”报告文件所在目录，但当前未对 .md 做统一展示或下载**，需要扩展。

### 2.2 Strix 报告样式与人类友好性

- **现有样式：**
  - **总报告**（`penetration_test_report.md`）：英文、Markdown；结构为四大节（Executive Summary / Methodology / Technical Analysis / Recommendations），层次清晰，符合常见渗透测试报告结构。
  - **单漏洞报告**（`vulnerabilities/vuln-xxxx.md`）：英文；包含 ID、Severity、Target、Endpoint、Method、CVE/CWE/CVSS、Description、Impact、Technical Analysis、Proof of Concept、Remediation 等，结构完整、层次分明。
- **对人类友好的不足：**
  - **语言**：全部为英文，若面向中文团队，需要中文描述。
  - **分散**：总报告与各漏洞分文件存放，缺少“一份完整文档”的阅读体验。
  - **格式**：已是 Markdown，层次性好，但若需“打印/归档/对外交付”，通常希望有一份**统一、可导出、语言与格式均友好**的渗透测试报告。

因此，**需要将 Strix 输出整合为一份统一的、面向人类阅读的渗透测试报告**，并在语言（中文）与层次性上做增强。

## 三、统一渗透测试报告方案：业界格式 + Strix 输出 + 人类友好

### 3.1 目标

- 将 **Strix 生成的总报告 + 所有漏洞详情** 合并为 **一份** 渗透测试报告文档。
- 满足：
  - **人类友好**：中文描述 + 良好的文本层次（标题层级、列表、表格、代码块保留）。
  - **业界通用结构**：参考一流渗透测试团队常见交付格式，与 Strix 的 Executive Summary / Methodology / Technical Analysis / Recommendations 及单漏洞结构对齐并扩展。

### 3.2 推荐的统一报告结构（结合 Strix 与业界实践）

建议生成的**统一报告**采用如下结构（可输出为单一 `.md`，或再转为 HTML/PDF 供下载或预览）。**实现时可结合 LLM**：由大模型完成执行摘要与方法论的中文翻译、漏洞描述与修复建议的润色与中文化、以及段落结构与表述优化，发挥大模型在语言与格式上的优势，提升报告可读性与专业性。

1. **封面 / 报告元信息**
   - 报告标题：如「安全渗透测试报告」
   - 目标系统 / 测试范围（来自 Strix 的 target 或本系统的作业/扫描目标）
   - 测试时间范围、报告生成时间
   - 执行方/工具说明（如 Strix + net-soc-ops）

2. **执行摘要（中文）**
   - 对应 Strix 的 **Executive Summary**，翻译或重写为中文。
   - 包含：整体风险判断、主要发现概览、业务影响简述、核心修复主题。

3. **测试范围与方法论**
   - 对应 Strix 的 **Methodology**，翻译或重写为中文。
   - 包含：测试类型（黑盒/灰盒/白盒）、遵循标准（如 OWASP WSTG）、范围说明、测试活动类别、约束说明（如不暴力破解）。

4. **漏洞发现总览**
   - 按严重程度（Critical / High / Medium / Low / Info）统计与列表。
   - 可来自 Strix 的 `vulnerabilities.csv` 或遍历 `vulnerabilities/*.md`，生成表格：序号、ID、标题、严重程度、对应章节/页码。

5. **漏洞详情（逐条）**
   - 对每个 `vulnerabilities/vuln-xxxx.md`：
     - 标题、ID、严重程度、发现时间；
     - 目标、端点、方法、CVE/CWE/CVSS（若有）；
     - 描述、影响、技术分析、复现步骤与 PoC、修复建议；
     - 保持原有层次结构，内容**翻译为中文**或提供中英对照（视需求选择）。

6. **技术分析总结**
   - 对应 Strix 的 **Technical Analysis**，翻译或重写为中文。
   - 可补充：漏洞之间的关联、根因归纳、技术栈与攻击面简述。

7. **修复与改进建议**
   - 对应 Strix 的 **Recommendations**，翻译或重写为中文。
   - 按优先级（立即/短期/中期）组织，便于客户落地。

8. **附录（可选）**
   - 术语表、参考标准、测试环境说明、报告版本与变更记录。

以上结构既保留了 Strix 的完整信息，又符合业界“执行摘要 → 范围与方法 → 发现列表 → 详细发现 → 总结与建议”的常见顺序，便于人类阅读与交付。

**功能范围（供 PRD 边界）：** 本阶段覆盖单次扫描对应一份统一报告；生成时机支持「按需生成」和/或「扫描完成后异步生成」；报告下载与在线预览。多扫描合并、PDF 导出、报告模板可配置、多语言切换等可列为后续迭代。

#### 3.3 数据来源与 Strix 输出约定

- **任务记录**：`StrixScanTask` 提供 `id`、`report_path`、`target_type`、`target_value`、`status`、`created_at`、`finished_at`、`job_execution_id` 等。`report_path` 为 Strix 报告目录（可能为 `.../strix_runs` 或其子目录，如 `.../strix_runs/172-18-40-99-8080_e44c`）。
- **总报告**：`report_path` 下（或其子目录下）的 **`penetration_test_report.md`**，英文，含 # Executive Summary / # Methodology / # Technical Analysis / # Recommendations 及正文。
- **漏洞列表与详情**：同目录下的 **`vulnerabilities/`**：**vulnerabilities.csv**（若存在，表头如 id, title, severity, timestamp, file）用于顺序与列表；**vulnerabilities/vuln-xxxx.md** 为每条漏洞的详情（元数据 + ## Description / Impact / Technical Analysis / Proof of Concept / Remediation）。
- **路径解析**：若 `report_path` 下无 `penetration_test_report.md`，则在其**子目录**中查找；漏洞顺序优先按 csv，无 csv 则按 severity 与 id 排序后遍历 .md。

#### 3.4 后端实现方案

**总体流程：**

1. **输入**：scan `task_id`。
2. **解析 report_path**：从任务记录取 `report_path`，解析得到实际包含 `penetration_test_report.md` 的目录（含子目录查找）。
3. **读取原始内容**：读取总报告全文并按标题切出四段；读取 vulnerabilities.csv 或遍历 vulnerabilities/*.md 得到有序漏洞列表及每条 .md 内容。
4. **拼接英文草稿**：按 3.2 结构将总报告四段与各漏洞 .md 拼成一份完整 Markdown（英文）。
5. **LLM 中文化与润色**（可选但推荐）：使用**系统管理配置的全局 OpenAI API Key**；对执行摘要、方法论、技术分析、修复建议及每条漏洞的描述/影响/复现/修复等段落分批调用 LLM 做中文翻译与表述优化；保留代码块与表格；代码块、CVE/CWE/CVSS 等可保留原文。
6. **输出与存储**：将最终 Markdown 写入如 `{report_path}/unified_penetration_test_report.md`；可选同时生成 HTML。可在任务表或单独表记录「统一报告已生成」及路径，供前端展示「下载」「预览」。
7. **失败与降级**：若未配置全局 Key 或 LLM 超时/限流，可降级为仅英文拼接或在报告中注明「未进行中文润色」，或返回错误提示配置 Key 后重试。

**接口设计（供 PRD 与开发对齐）：**

| 接口 | 方法 | 说明 |
|------|------|------|
| 生成统一报告 | `POST /api/config-module/strix/scans/{task_id}/unified-report` | 触发生成（读取→拼接→可选 LLM 润色→落盘）。返回 202/200，body 可含 `unified_report_path` 或 `ready`。异步时需轮询或回调。 |
| 获取统一报告（下载） | `GET /api/config-module/strix/scans/{task_id}/unified-report` | 已生成则返回文件流（.md 或 .html），设置 Content-Type、Content-Disposition；未生成可 404/409 并提示先调用生成。 |
| 获取统一报告（预览） | `GET /api/config-module/strix/scans/{task_id}/unified-report?format=html` 或单独 path | 已生成 HTML 则返回 HTML 供 iframe/新标签页；无则 404。 |

现有 `GET /scans/{task_id}/report` 保留（仅 .html）；统一报告通过以上新接口暴露。前端「下载报告」可优先引导「下载统一报告」，未生成时回退「下载原始报告」（或扩展支持返回 penetration_test_report.md）。

#### 3.5 前端行为（供 PRD 与交互说明）

- **渗透测试报告列表/详情页**：每条扫描展示「生成统一报告」（未生成时）或「下载统一报告」「预览」（已生成时）。下载调用 `GET .../unified-report` 保存为 .md/.html；预览为新开标签页或前端 Markdown 渲染。
- **错误与状态**：未配置全局 OpenAI Key 时提示「请先在系统管理中配置 OpenAI API Key 以启用报告中文化」；LLM 失败时展示降级说明或重试入口。

#### 3.6 API Key 配置策略（系统全局 vs 渗透测试）

- **系统管理页面：全局 OpenAI API Key**
  - 在**系统管理**中提供 **OpenAI API Key**（或「全局 LLM API Key」）配置项，供 **netops 全局功能**使用。
  - 使用场景包括但不限于：统一渗透测试报告的**中文化与润色**（见 3.3）、后续可能接入的其它需调用大模型的能力（如配置摘要、工单摘要、告警解释等）。
  - 该 Key 与渗透测试执行时使用的 Token **相互独立**，便于权限与配额隔离。

- **渗透测试功能：沿用渗透测试组件 API Token**
  - **渗透测试**相关能力（Strix 扫描执行、Agent 推理等）**不使用**系统管理中的全局 OpenAI Key。
  - 继续使用 **「渗透测试组件」**（原子功能 → 渗透测试）页面中配置的 **API Token / LLM 配置**（即 Strix 的 `STRIX_LLM`、`LLM_API_KEY`、`LLM_API_BASE` 等），保证：
    - 渗透测试与其它业务在使用模型与计费上隔离；
    - 运维可单独为 Strix 配置专用模型或自建 API，而不影响全局功能。

#### 3.7 依赖与配置

- **系统管理 · 全局 OpenAI API Key**：须在系统管理页提供配置项（如 OpenAI API Key 或「全局 LLM API Key」），后端可安全读取。统一报告的 LLM 中文化**仅使用该 Key**，与渗透测试 Strix 用 Key 隔离（见 3.6）。
- **Strix 输出**：依赖 Strix 在 `report_path`（或其子目录）下生成 `penetration_test_report.md` 与 `vulnerabilities/*.md`（及可选 vulnerabilities.csv）。若 Strix 版本或配置导致路径/文件名变化，需在实现中做兼容。

#### 3.8 非功能要点（PRD 可引用）

- **性能**：报告生成可能耗时较长（尤其启用 LLM），建议异步生成 + 轮询或完成通知；或同步但设合理超时与前端 loading。
- **安全**：统一报告可能含敏感内容，下载与预览需与现有权限一致；全局 API Key 仅后端使用，不暴露前端。
- **存储**：统一报告可随 `report_path` 存放或集中按 task_id 命名；需考虑磁盘占用与定期清理（可与 Strix 工作区清理策略一致）。

#### 3.9 验收要点与可交付物（供 PRD 验收标准）

- 给定已完成的 Strix 扫描任务（含 report_path 及 penetration_test_report.md、vulnerabilities/），能**触发生成**并**下载**符合 3.2 结构的**单一文档**（.md 或 .html）。
- 文档中执行摘要、方法论、技术分析总结、修复建议及漏洞详情的自然语段为**中文**（在已配置全局 OpenAI Key 且 LLM 成功时）。
- 文档层次清晰（标题、表格、代码块保留），可直接用于阅读或对外交付。
- 渗透测试报告列表/详情页具备「生成统一报告」「下载统一报告」「预览」入口，与现有「下载报告」区分明确。
- 未配置全局 Key 或 LLM 失败时，有明确提示或降级（如仅英文报告），不阻塞报告下载。

### 3.10 小结

- **账号密码**：在**渗透测试节点**增加可选“测试账号”“测试密码”，并在 UI 明确提示“非静态测试时填写可提高测试结果；仅用于授权环境”；后端将其安全拼入 `instruction` 传给 Strix。
- **报告读取**：工程能读到 Strix 报告目录与 .md 内容，但当前下载接口只认 .html，需扩展以支持 .md 或统一报告。
- **人类友好统一报告**：将 Strix 的 `penetration_test_report.md` 与 `vulnerabilities/*.md` 合并为一份报告，采用 3.2 业界常用结构；**结合 LLM** 做中文化与表述优化，输出为单一文档；实现细节见 3.3～3.9（数据来源、后端流程与接口、前端行为、API Key、依赖、非功能、验收）。
- **API Key 策略**：系统管理页配置**全局 OpenAI Key** 供 netops 其它功能（含统一报告中文化）使用；**渗透测试**单独使用渗透测试组件提供的 API Token，与全局 Key 隔离（见 3.6）。
