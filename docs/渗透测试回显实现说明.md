# 渗透测试回显实现说明

本文档说明 Strix 渗透测试任务中「回显」的数据来源、写入方式、读取接口及性能与安全设计。

---

## 1. 回显的含义与数据流

**回显**指用户能看到的扫描过程与结果输出，分为两类：

| 阶段     | 内容说明                         | 数据来源                    |
|----------|----------------------------------|-----------------------------|
| 运行中   | 漏洞数、Agents、Tools 等进度指标 | 磁盘文件 `progress.json`    |
| 结束后   | 完整 stdout / stderr 文本       | 数据库 `strix_scan_tasks.summary` |

整体数据流：

```
Strix 进程 stdout/stderr
        │
        ├── 运行中：按行读取 → 正则解析数字 → 写入 progress.json → GET /scans/{id}/progress 读
        │
        └── 结束后：完整拼接 → 写入 task.summary（DB）→ GET /scans/{id} 返回（截断展示）
```

---

## 2. 运行中回显：progress.json

### 2.1 为何需要 progress.json

- 任务由**作业执行**创建时，后端用 `Popen` 跑 shell 脚本（`run_strix.sh`），脚本内部执行 `strix` CLI。
- 若用 `proc.communicate()` 一次性等进程结束再拿 stdout，运行期间前端无法拿到任何进度。
- 因此改为**线程按行读 stdout**，从中解析出数字并**实时写入**任务目录下的 `progress.json`，前端轮询 progress 接口即可拿到「当前漏洞数 / Agents / Tools」。

### 2.2 写入方（生产者）

有两处会写 `progress.json`，逻辑一致：

1. **作业执行路径**：`netops-backend/app/services/job.py`  
   - 对每个渗透节点：`Popen` 执行 `run_strix.sh`，`cwd=run_dir`（例如 `data/strix_workspace/job_45_0`）。
   - 启动两个线程：
     - `read_stdout(pipe)`：`for line in iter(pipe.readline, "")` 逐行读 stdout，将行追加到 `stdout_lines`；每次用当前全部内容 `buf = "".join(stdout_lines)` 调用 `_parse_stdout_stats(buf)`；若解析到新数字则更新 `latest_stats` 并**写盘** `progress_file = run_dir/progress.json`。
     - `read_stderr(pipe)`：仅收集 stderr 行，不写 progress。
   - 进程结束后再用 `stdout_lines`/`stderr_lines` 拼成完整字符串写入 `task.summary`。

2. **API 创建任务路径**（若仍使用）：`netops-backend/utils/strix_runner.py` 的 `run_strix_sync()`  
   - 同样用线程读 stdout，解析 `_parse_stdout_stats` 并写入 `cwd/progress.json`（`cwd = workspace_dir/run_name`）。

**解析规则**（`strix_runner._parse_stdout_stats`，job 复用该函数）：

- 在整段 stdout 文本中用正则提取：
  - `Vulnerabilities\s+(\d+)` → `vulnerabilities`
  - `Agents\s+(\d+)` → `agents`
  - `Tools\s+(\d+)` → `tools`
- 每次解析到新键则更新字典，再整体写入 `progress.json`，例如：`{"vulnerabilities": 2, "agents": 1, "tools": 3}`。

### 2.3 读取方（消费者）

- **接口**：`GET /api/config-module/strix/scans/{task_id}/progress`
- **实现**：`netops-backend/routes/strix_integration.py` 中 `get_scan_progress()`：
  1. 查库得到 `task.run_name`（如 `job_45_0`）与 `task.status`。
  2. 仅当 `status in ("running", "pending")` 时使用**内存缓存**（同一 `task_id` 在 2 秒内直接返回上次结果），减少重复读盘。
  3. 未命中缓存时：`workspace_root = backend/data/strix_workspace`，拼路径 `workspace_root/run_name/progress.json`，做**路径校验**（必须落在 `workspace_root` 下），再读文件并解析 JSON，返回 `{ "vulnerabilities", "agents", "tools" }`；文件不存在或异常则返回三个 `null`。

## 3. 结束后回显：summary（DB）

### 3.1 写入

- **作业路径**：在 `job.py` 中，`proc.wait()` 返回后（或超时/异常收尾后），用 `stdout_lock`/`stderr_lock` 下的 `stdout_lines`/`stderr_lines` 拼成 `full_stdout`、`full_stderr`，再：
  - `task.summary = json.dumps({ "stdout": full_stdout[:STRIX_SUMMARY_MAX_CHARS], "stderr": full_stderr[:STRIX_SUMMARY_MAX_CHARS] })`（库内单字段最大 1000K 字符）。
- **API 路径**：`run_strix_sync` 返回的 stdout/stderr 在 `strix_integration._run_scan_task` 里以同样方式写入 `task.summary`。

### 3.2 读取与展示截断

- **接口**：`GET /api/config-module/strix/scans/{task_id}`（详情）、`GET /api/config-module/strix/scans`（列表）。
- **逻辑**：
  - 从 DB 读出 `task.to_dict()`，其中 `summary` 为完整 JSON（可能很大）。
  - 返回前经 `_trim_summary_for_display(summary)`：对 `summary.stdout`、`summary.stderr` 做**尾部截断**，单字段最多 `SUMMARY_DISPLAY_MAX_CHARS`（当前 1K 字符），避免单次接口返回数 MB 的 JSON，减轻网络与前端压力；**库内仍保存完整内容**。

## 4. live_echo.txt 说明（当前未使用）

### 4.1 当前行为

- **运行中**：代码**不再写入** `live_echo.txt`。为减少磁盘 IO 和“实时同步”带来的开销，只更新 `progress.json`（模型、漏洞数、Agents、Tools 四项），完整输出仅在任务结束时写入 DB 的 `task.summary`。
- **接口**：`GET /api/config-module/strix/scans/{task_id}/echo` 仍存在，会读 `run_dir/live_echo.txt`（若存在）；新任务不会生成该文件，故接口通常返回空。历史任务若曾写过该文件，仍可被读取。

### 4.2 若需要“回显文件”的推荐写法

若以后要恢复写 `live_echo.txt`（例如供某处“查看完整回显”用），建议**仅在任务结束时写一次**：

1. **触发时机**：任务结束（成功/失败/超时/取消）时，在写入 `task.summary` 之后。
2. **写入方式**：先**清空**再写——`open(live_echo_path, "w")` 后写入完整的 `full_stdout + full_stderr`（或分两段写），**不要**在运行中按块 append。
3. **原因**：运行中 append 会导致 Strix TUI 每次刷新同一块界面都追加一段，文件里出现大量重复的框和相似内容；只在结束时写一份完整内容，则无重复、便于查看和排查。

## 5. 性能与安全要点

| 项         | 做法 |
|------------|------|
| 运行中读盘 | progress 接口仅对 running/pending 任务做 2 秒内存缓存，同一 task_id 在 TTL 内不重复读 `progress.json`。 |
| 接口体积   | 列表/详情返回的 `summary` 中 stdout/stderr 仅返回尾部 1K 字符；完整回显仍在 DB。 |
| 路径安全   | 读 `progress.json` 时用 `os.path.realpath` 与 `workspace_root` 比较，确保不读工作区外路径。 |
| 存库上限   | `task.summary` 单字段（stdout/stderr）最大 1000K 字符（PRD 约定）。 |

## 5. 涉及文件与 API 一览

| 文件/接口 | 作用 |
|-----------|------|
| `app/services/job.py` | 作业执行渗透：Popen + 双线程读 stdout/stderr，写 `run_dir/progress.json`，结束后写 `task.summary`；复用 `strix_runner._parse_stdout_stats`。 |
| `utils/strix_runner.py` | `_parse_stdout_stats()`：从 stdout 文本解析 vulnerabilities/agents/tools；`run_strix_sync()`：线程读 stdout 并写 progress.json（供 API 创建任务用）。 |
| `routes/strix_integration.py` | `get_scan_progress()`：读 progress.json，带 2 秒缓存与路径校验；`get_scan()`/列表：返回前对 summary 做 `_trim_summary_for_display()`；`_parse_strix_stdout_stats()`：详情中从 summary.stdout 解析 strix_stats。 |
| `GET /api/config-module/strix/scans/{id}/progress` | 返回运行中进度：`{ vulnerabilities, agents, tools }`。 |
| `GET /api/config-module/strix/scans/{id}` | 返回任务详情，含截断后的 `summary` 及从 summary 解析的 `strix_stats`。 |

---

## 7. 小结

- **运行中回显**：Strix 进程的 stdout 被**按行读取**，经**正则解析**得到 Vulnerabilities/Agents/Tools，**实时写入**任务目录下的 `progress.json`；前端轮询 progress 接口即可展示进度；接口对 running/pending 任务做短期缓存以减轻读盘。
- **结束后回显**：完整 stdout/stderr 写入 `task.summary`（DB），列表/详情返回时对 stdout/stderr 做**尾部 1K 截断**再返回，既保证可排查问题又避免单次响应过大。
